# Changelog

### Unreleased

- **MQTT broker TLS.** Transport TLS (`mqtts`, CA bundle, cert validation) is not implemented on `mqtt_publisher.py` (plain TCP only). Until then use plain port 1883 or a LAN broker without TLS.

### v0.7.7 (July 2026)

First tagged release of the javastraat/meshpoint fork: LoRaWAN sniffing, multi-radio capture, the RTL-SDR web listener, and a dashboard self-update repair -- plus everything from upstream's own v0.7.7 (backup/restore, RF Environment tab, mesh broadcast cadence, operator tools; merged, bullets below). Run `install.sh` when upgrading: new sudoers rules for backup restore. **Upgrade note:** boxes on v0.7.6 cannot fetch this release from the dashboard (that is the bug being fixed); one manual round on the gateway is required: `cd /opt/meshpoint && sudo git fetch origin main && sudo git reset --hard origin/main && sudo systemctl restart meshpoint`. The restart installs the corrected sudoers rules; Check/Apply works from the dashboard again afterwards.

#### LoRaWAN sniffing (SX1302)

- **Passive LoRaWAN capture on EU868.** Five real 125 kHz LoRaWAN channels (867.9–868.7 MHz, sync word 0x34) on SX1302 ch0–ch4 while Meshtastic stays on the 250 kHz service channel (869.525 MHz, 0x2B) — dual sync word via direct service-channel register writes after `lgw_start()`.
- **LoRaWAN MAC decoder.** Join-Request, Data Up, and Rejoin frames parsed (DevEUI/AppEUI, DevAddr, FCnt, FPort, MIC); payloads stay encrypted (no session keys), listen-only.
- **LoRaWAN dashboard page.** Devices, recent packets, and stats views backed by `GET /api/lorawan/devices`, `/api/lorawan/packets`, `/api/lorawan/stats`.
- **Strict isolation.** LoRaWAN traffic is never relayed and never enters the mesh node roster or telemetry store; router checks LoRaWAN before Meshtastic to avoid false positives.
- **CSV export on every protocol page.** An Export button on the LoRaWAN, Meshtastic, and MeshCore pages downloads the active tab's dataset — all captured packets, or the full device/node/contact census (not just the last 100 on screen). Streamed row-by-row from the database so a full-table packets export never buffers in memory; UTF-8 with a BOM so Excel renders the emoji-heavy MeshCore names; packet exports carry the decoded payload as a JSON cell, and LoRaWAN additionally flattens DevEUI/FCnt/FPort/MIC into their own columns for coverage and frame-counter-gap analysis. Six new `GET /api/{protocol}/export/{packets,census}.csv` endpoints (viewer-open, authenticated by the session cookie).
- **FPort/FCnt show up in the packets log.** The LoRaWAN page's FPort and FCnt columns were permanently "--": the API read the decoded payload with underscored key names (`f_cnt`) while the decoder stores them without (`fcnt`). The frame counter and port now display for every data packet.
- **Service channel RF chain fix.** ch8 now picks the correct radio chain from its frequency instead of hardcoded RF0.

#### Multi-radio capture (5 networks at once)

- **Multiple MeshCore USB companions.** `capture.meshcore_usb` accepts a list (up to 4) with per-stick labels; packets carry `meshcore_usb_<label>` as capture source. New `PUT /api/config/capture/meshcore-companions` replaces the list atomically; the Configuration page grew a dynamic companion section.
- **Multiple Meshtastic USB sticks.** `capture.serial` accepts the same list-plus-label shape as `meshcore_usb` (e.g. one stick on 433 MHz, one on 868 MHz), packets tagged `serial_<label>`; the legacy single `serial_port`/`serial_baud` fields keep working unchanged when the list is empty. Also fixes a latent duplicate-capture bug: meshtastic-python publishes every open USB interface's packets on one process-wide pypubsub topic, so running more than one stick without an identity check would have cross-attributed packets between devices. New Configuration → Serial page edits this list from the dashboard (add/remove/label, same shape as the MeshCore companion editor) instead of hand-editing `local.yaml`; new `PUT /api/config/capture/serial-devices`.
- **Serial Meshtastic capture crash and silent packet loss fixed.** The first time a `serial` USB source actually ran, it crashed the pipeline (`TypeError: object of type 'MeshPacket' has no len()`) — meshtastic-python always hands back the raw protobuf object, not bytes, so the old code never fell back to header reconstruction. Fixed, plus two bugs the crash was hiding: the reconstruction fallback read the wrong dict key (`encoded`/hex instead of the real `encrypted`/base64), and it required a `decoded` key that is absent by design on exactly the packets a passive sniffer cares about most — those the connected stick's own key couldn't decrypt (traffic on a channel it isn't configured for). Those were silently dropped instead of being handed to Meshpoint's own decoder/`channel_keys`.
- **Serial Meshtastic packets the connected stick decrypts itself now show their real type.** `encrypted`/`decoded` share one protobuf oneof on MeshPacket, so a packet the stick's own key successfully decrypted locally has no ciphertext left at all — these always showed as "Unknown" even though the decoded content (position, telemetry, nodeinfo, text) was sitting right there in the packet dict, discarded before Meshpoint ever saw it. `RawCapture` gained an optional `pre_decoded` field (portnum + inner payload bytes); `MeshtasticDecoder.decode()` dispatches it directly instead of attempting (and failing) to decrypt an empty body, so these packets now decode to their real `PacketType` just like concentrator-captured ones.
- **Band tag on node cards and the drawer.** Nodes now show a "433 MHz" / "868 MHz" chip derived from their latest packet's labelled capture source (`serial_433`, `meshcore_usb_868`, `concentrator`, …), so a 433/868 multi-stick setup is easy to tell apart at a glance.
- **Real region/frequency/SF/bandwidth/coding-rate for `serial` Meshtastic capture, plus a topbar badge per USB stick.** The `serial` source previously stamped every packet with a hardcoded `906.875 MHz` (a US915 placeholder) and SF11/BW250 regardless of the connected stick's actual region or preset. It now reads the stick's real config at connect time (meshtastic-python completes this handshake automatically before the connection call returns) — region, channel, primary channel name, modem preset (or the raw custom SF/BW/coding-rate when not using a named preset) — the same "read once at connect, mirrors MeshCore's `self_info` fix" pattern. New `src/radio/channel_frequency.py` computes the actual operating frequency using the same formula as `meshtastic/firmware`'s `RadioInterface.cpp` (verified against a local firmware checkout and cross-checked against a real device's observed frequency): region band, modem bandwidth, and — for the default/auto channel — a djb2 hash of the primary channel name (or the modem preset's display string when the channel name is blank), exactly reproducing the firmware's own frequency-slot selection instead of a fixed per-region guess. An explicit non-default channel, or one outside the region's valid slot range, reports `0.0` (this codebase's existing "unknown" sentinel) rather than a wrong number. New topbar badge — one small cyan chip per configured Meshtastic USB stick, laid out the same as the Meshtastic and MeshCore chips (brand, lamp, the stick's own node ID as the "call sign" slot, region, frequency, and its modem preset) — separate from the single "primary" MeshCore companion chip, since every serial device is capture-only (no TX binding to pick "the" one); reacts to a dashboard websocket drop the same way the other two chips do (amber "reconnecting" lamp, region/frequency/preset blanked) instead of silently keeping stale data on screen through a restart. New `scripts/backfill_serial_meshtastic_signal.py` corrects old rows still carrying the `906.875` placeholder.
- **Serial Meshtastic sticks no longer capture their own self-telemetry as spammy, confusing packets.** A connected stick's own periodic self-report (battery/uptime NodeInfo/Telemetry) passes through the same event stream as genuinely received packets — meshtastic-python doesn't distinguish them — and firmware's own convention (`rx_rssi == rx_snr == 0` meaning "generated locally, never actually received over the air") confirmed these had no real signal to show; they were landing in the packet feed and node counters as a confusing `-100 dBm` reading of the stick reporting on itself, roughly once a minute. The connect-time handshake now also reads the stick's own node number (`interface.myInfo.my_node_num`, no config needed) and drops any packet it originates before decode/storage/the feed/counters ever see it. The detected ID is exposed via the API and shown in the topbar badge's tooltip for visibility. New `scripts/purge_self_originated_node.py` removes historical packet/telemetry/message/node rows left over from before this fix (the drop only stops future capture, it doesn't retroactively clean the database). The drop now exempts `TEXT_MESSAGE_APP`: a chat message typed via a BLE/WiFi-connected app on the same physical stick is genuine content the user wants in the Messages panel, not a beacon, so it still reaches decode/storage even though the stick is its own "from" node for that packet — only routine self-reported telemetry/nodeinfo/position stays dropped.
- **Labelling a MeshCore companion no longer silently disables its TX/advert/status.** The dashboard's "MeshCore connected" lookup matched capture sources by exact name (`"meshcore_usb"`), which breaks the moment a companion gets a label (its name becomes `meshcore_usb_<label>`) — no crash, just a topbar stuck on "No companion" and no startup advert. Fixed to the same `startswith` rule already used for packet routing. Receive/capture was never affected, only transmit and the live-status chip.
- **MeshCore signal metadata.** Frequency/bandwidth/SF stamped from each companion's connect-time radio handshake (per stick, so 868 and 433 report their real channel), and hop count derived from `path_len` on contact/channel messages.
- **Simultaneous LoRaWAN + Meshtastic 868 + MeshCore 868/433 + Meshtastic 433.** Concentrator, two labelled MeshCore sticks, and a Meshtastic serial node run side by side; only Meshtastic is ever relayed.

#### Hardware page and band spectrum

- **Band spectrum card on the RF Environment page.** The SX1302's spectral scanner now sweeps the whole region band (one scan per 100 kHz, EU868: 863–870 MHz) every 5 minutes and draws the result as a spectrum chart — median and peak (p95) level per step — with the LoRaWAN, Meshtastic, and MeshCore channel positions overlaid as markers. Includes an admin "Sweep now" button and hover readout; sweep cadence is configurable via `radio.spectrum_sweep_interval_seconds` (0 = on-demand only). Packet capture continues during sweeps. While the first sweep is still running the card shows a placeholder and fills in automatically instead of hiding.
- **Spectral-scan percentiles fixed on descending-histogram HALs.** The M1's libloragw returns the spectral-scan histogram with levels in descending order, which flipped the percentile walk — the sweep occasionally reported peak below median, and the noise-floor pill computed its floor from the wrong end of the histogram. Levels are now sorted before the walk, so floor ≤ median ≤ peak always holds.
- **Concentrator channel plan visible on the Hardware page.** A new read-only card lists all 9 SX1302 slots — frequency, bandwidth, SF, sync word, protocol, RF chain, and state per channel (EU868: ch0–ch4 LoRaWAN sniffing at 867.9–868.7 MHz with sync 0x34, ch5–ch7 disabled, ch8 Meshtastic at 869.525 MHz with sync 0x2B) — so the full multi-protocol setup is finally self-explanatory from the dashboard.
- **Hardware page: Meshtastic settings unified in one card.** The "Radio Configuration" card is now "Meshtastic Configuration" and includes a compact channel list (matching the MeshCore card's style); the separate Channels card is gone. The computed values (SF, BW, CR, sync word, preamble) render as regular label/value rows in the same grid as region and frequency instead of a separate tile strip. The MeshCore Companion card top switched to the same label/value rows as the Meshtastic card, and NodeInfo Broadcast moved below the companion card.

#### Stats and node insights

- **SNR distribution chart on Stats.** The Signal Intelligence section pairs the RSSI histogram with an SNR histogram (2 dB buckets over the last 500 packets), so noise-margin health is visible at a glance.
- **Recent packets in the node drawer.** Opening a node now shows its last 15 packets (time, type, RSSI/SNR to one decimal) in a collapsible section, next to the existing metrics history; works for Meshtastic and MeshCore nodes alike.
- **Near-field packets no longer skew signal stats.** RSSI readings above −20 dBm (a node centimetres from the antenna during desk testing, or 0-dBm placeholders) are excluded from the Stats best/avg RSSI tiles and the RSSI histogram, so "Best RSSI" reflects actual reach.
- **Network totals no longer capped at 500 nodes.** The network summary (Stats page and `meshpoint report`) computed its totals over the 500 most recent nodes, silently under-reporting node counts, positions, and packet sums on larger installs; it now uses whole-table SQL aggregates.
- **Total Packets tile shows 24h / total.** The dashboard stat card now reads like the Nodes Discovered tile ("339 / 935"), pairing the last 24 hours with the all-time count.

#### Configuration and server

- **Quieter boot log.** The MeshCore contact sync logged the companion's entire roster — 350 lines per boot on a well-connected stick. It now logs the peer count, the first 10 names, and "… and N more"; the full roster stays browsable on the MeshCore page.
- **Startup banner: one line per capture source.** The banner's single Source/Frequency pair (which only showed the concentrator's Meshtastic channel) is replaced by a per-source list with each radio's real plan: the concentrator's LoRaWAN span + Meshtastic service channel, each MeshCore companion's frequency/SF from its connect handshake, and each Meshtastic USB stick's resolved frequency and region. Sources still handshaking show "(radio info pending)". The logo box's right border also lines up with its corners again.
- **Web server port is now set in the config.** The dashboard bind address comes from `dashboard.host` / `dashboard.port` in the YAML config (override in `local.yaml`, applied on service restart) via the new `src/serve.py` launcher; previously the port was hardcoded in the systemd unit and the config value only affected the startup banner. If the config fails to load, or the configured address cannot be bound (port in use, privileged port, bad host), the server falls back to `0.0.0.0:8080` so the dashboard (and its update/rollback page) stays reachable.
- **Relay burst and RSSI filters editable in Configuration → Transmit.** Burst size and the min/max RSSI relay window (only packets heard inside the window are rebroadcast) can now be set from the dashboard; previously these existed in `local.yaml` and the relay API but had no UI.

- **Unused API endpoints removed.** `GET /api/packets/protocols`, `/api/packets/types`, `/api/nodes/map`, and `/api/telemetry/{id}` (+`/history`) served data already available through `/api/stats/summary`, `/api/nodes`, and `/api/nodes/{id}/metrics_history`; nothing in the dashboard or CLI used them.

#### SenseCap M1 hardware

- **SenseCap M1 onboard LED/button/fan GPIO probe.** No public schematic exists for this board's expansion peripherals; new `scripts/test_gpio_hardware.py` (`gpiozero`, not a runtime dependency) identifies and exercises them. `led`/`button`/`fan` test one pin directly; `button-scan`/`fan-scan` sweep a batch of candidate pins at once (skipping the concentrator's SPI0 bus, the I2C crypto/temp chips, the HAT ID pins, and the concentrator reset lines) rather than guessing pin-by-pin. Confirmed live on real hardware: LED = GPIO 22, button = GPIO 27, fan = GPIO 13.
- **Temperature-driven PWM fan control.** New opt-in `fan:` config section drives the SenseCap M1's onboard fan from CPU temperature (`/sys/class/thermal/thermal_zone0/temp`) instead of running it flat-out or not at all. GPIO 13 is a hardware-PWM-capable pin on the Pi 4 (BCM2711 PWM1), so speed ramps linearly between `min_temp_c`/`max_temp_c` rather than a blunt on/off relay; a `hysteresis_c` gap and a `min_duty` floor (most small fans stall below some duty rather than spinning slower) avoid chatter and stalling right at the threshold. New `src/hardware/fan_control.py` (`FanCurve` for the pure duty-cycle math, `FanController` for the polling loop), runs as a background task for the app's lifetime same as the noise-floor emitter. Disabled by default — this hardware doesn't exist on RAK V2/Chameleon/DIY builds. `fan: {enabled: true}` alone is sufficient; every other key has a matching default, also documented (with inline comments) in `config/default.yaml` so the full set of knobs is discoverable without reading source. Caught before release: `_apply_yaml`'s `section_map` (the dict `local.yaml`/`default.yaml` sections get merged through) is manually maintained rather than derived from `AppConfig`'s fields, and `fan` was missing from it — the whole section was being silently ignored, so `enabled: true` would have done nothing at all. Fixed, with a regression test asserting `fan:` in `local.yaml` actually reaches `cfg.fan`. New `gpiozero`/`lgpio` runtime dependencies (`requirements.txt`) — `lgpio` specifically because gpiozero's pure-Python fallback pin factory (used when no real GPIO backend is installed) only allows PWM on pins in its hardcoded per-Pi-model table, and rejects a custom carrier board's repurposed GPIO with `PinPWMUnsupported` even though it's a genuine PWM-capable SoC pin; `lgpio` doesn't have that restriction. `FanController` now detects that specific exception and logs the `pip install lgpio` fix directly instead of a raw traceback. Live-verified the full install chain on a real Bookworm/trixie SenseCap M1: `pip install lgpio` builds a C extension with no piwheels wheel available, needing `python3-dev` + `swig` (build tools) and `liblgpio-dev` (the native `liblgpio.so` the extension links against — `cannot find -llgpio` without it) installed via `apt` first; documented as one block in `TROUBLESHOOTING.md`. One more gap surfaced after that: `lgpio` creates its notification pipe directly in `WorkingDirectory` (`/opt/meshpoint`), but `meshpoint.service`'s `ExecStartPre` only ever chowned `config/` and `data/` to the `meshpoint` service user, never the top-level directory — so it still fell back to the PWM-incapable pin factory with `xCreatePipe: ... Operation not permitted`. Fixed in `meshpoint.service` (`ExecStartPre` now hands the whole install tree to the service user recursively — see the self-update bullet below for why recursive), self-healing on every start regardless of how the checkout was deployed. Live-verified working end to end: `Fan control started on GPIO13` with no exception. The duty-change log line stays at `debug` (it was temporarily `info` to verify the ramp live, but with normal temperature jitter it fires on most 10-second polls — thousands of journal lines a day); the dashboard's Fan card is the live view. New stats-bar **Fan** card (after Load Avg) shows the current duty percentage and what it changed from (`"was 35%"`), hidden entirely when fan control isn't enabled; backed by new `fan_duty_percent`/`fan_previous_duty_percent` fields on `GET /api/device/metrics`, sourced from `FanController.current_duty`/`previous_duty` (only updated on an actual change, not every poll, so "previous" reflects the last real transition). Live logs then showed `(was X)` values disagreeing with the previously printed duty (and "changes" logged when nothing visibly changed): the controller compared the freshly computed full-precision duty against the pin's read-back value, which `lgpio` quantizes differently. It now rounds the duty to the 1% steps the log and dashboard display and compares against its own last-set value, so consecutive log lines agree and identical polls stay silent.
- **User button.** New opt-in `button:` config section gives the SenseCap M1's case button (GPIO 27, confirmed via the GPIO probe script) two gestures. Short press: advert on every TX-capable radio — concentrator Meshtastic NodeInfo, MeshCore companion advert, and each Meshtastic USB stick broadcasting its own NodeInfo — serialized 2 s apart because the two 868 MHz signals overlap outright (869.525/BW250 contains 869.618/BW62.5) and same-box TX deafens the neighbouring receiver; a 30 s cooldown stops advert spam, and LoRaWAN stays out by design (pure listener). Hold 3 s: restart the meshpoint service — the recovery that works when the dashboard doesn't; fired detached so it survives its own process dying, sudoers rule already shipped. The status LED narrates every gesture (ack double-blink, cooldown-denied long blink, fast-blink hold warning), and the controller boots disarmed so a button held straight through the restart it triggered can never re-trigger. Poll-based state machine, same testable pattern as the fan and LED. New `src/hardware/button_control.py`; documented in `config/default.yaml` and `docs/CONFIGURATION.md`.
- **Status LED.** New opt-in `led:` config section drives the SenseCap M1's onboard case LED (GPIO 22, confirmed via the GPIO probe script) as a glanceable status light with four states: steady on = service running and every configured capture source healthy; a brief off-flicker per captured packet (a router-style traffic dip — `activity_blink: false` for a calm steady light); 1 Hz blink = one or more capture sources down; dark = service not running (free — the kernel releases the GPIO line when the process dies). Plain on/off GPIO, deliberately no PWM, so it works even without `lgpio`. Health reads the same per-source running flags the capture pipeline maintains, and packet activity polls the existing stats counter at 10 Hz — the capture hot path is untouched. New `src/hardware/led_status.py`, documented in `config/default.yaml` and `docs/CONFIGURATION.md`.

#### MeshCore repeaters

- **MeshCore repeater monitoring.** New opt-in `repeater_poll:` config section periodically queries MeshCore repeaters you operate for their status and telemetry — the same `req_status`/`req_telemetry` calls the phone app and `meshcore-cli` make, through the companion Meshpoint already holds. MeshCore nodes advertise identity only (unlike Meshtastic's broadcast telemetry), so this fills the otherwise-empty repeater stats: battery, uptime, airtime, packet counters (flood/direct), noise floor, and LPP sensors (temp/humidity/pressure). ### v0.7.3.1 (May 13, 2026)

Hotfix on top of v0.7.3 the same day. Reported by Willard on Discord ~3h after release: dashboard stuck on "Reconnecting..." with no data after upgrading. Two compounding bugs in the new auth path; a stale browser tab against an auth-required server is enough to trigger both.

- **WS auth close frame now actually reaches the browser.** `src/api/server.py` was calling `await websocket.close(code=4401)` *before* `await websocket.accept()`, which causes Starlette to fail the WebSocket handshake with HTTP 403 instead of completing the handshake and sending a close frame. Browsers translate that to close code `1006` (Abnormal Closure) on the JS side, and our `frontend/js/websocket_client.js` only special-cases `4401` for the redirect-to-/login path -- so unauthenticated WS connections fell through to the generic reconnect loop and stuck forever. Fix: `accept()` first, then `close(code=WS_AUTH_CLOSE_CODE)`. Validated end-to-end on .141 (cookie cleared, dashboard refreshed → bounces to /login as intended).
- **Dashboard root (`/`) now redirects unauthenticated requests to `/login`.** The `StaticFiles(directory=..., html=True)` mount on `/` was serving `index.html` to everyone with no auth check, so a stale browser tab could load the new SPA JS and immediately fight the now-auth-required `/ws`. New explicit `@app.get("/")` route registered ahead of the static mount: 302s unauthenticated requests to `/login` (or `/setup` if no admin password is set yet), serves `index.html` for valid sessions. Static asset paths (`/css`, `/js`, `/assets`, etc.) still fall through to the existing mount.
- **Client defense in depth.** `frontend/js/websocket_client.js` now tracks whether the socket reached the `open` state. If `onclose` fires *without* a prior `onopen`, the handshake failed before the close frame could be delivered. The client probes an auth-gated endpoint (`/api/device/status`); the global 401 interceptor in `app.js` then handles the redirect if it's an auth-shaped failure, while real network blips fall through to the existing reconnect schedule. Belt-and-suspenders coverage so a future server-side regression in the close-code path can't strand users again.
- **Internal:** new `tests/test_dashboard_root_route.py` (unauthenticated GET `/` → 302 to `/login`; GET `/` with no admin password yet → 302 to `/setup`; GET `/` with valid cookie → 200 + index.html). New `tests/test_websocket_auth_close_code.py` asserts the WS handshake completes and the close frame's code is exactly `4401` for both no-cookie and bad-cookie cases (regression for the `accept()`-before-`close()` requirement). 403 tests passing, ruff clean.

MeshCore reliability patch. Small follow-up to v0.6.5 cleaning up rough edges around the MeshCore USB companion. No edge concentrator changes, no cloud changes.

Previously the UI showed the conversation key (`broadcast:meshtastic:0`) in place of the sender because the backend never resolved the source node for Meshtastic broadcast text packets. The v0.6.2 sender-name fix only covered MeshCore; this finishes the job for Meshtastic. ([#19](https://github.com/KMX415/meshpoint/issues/19))
- **Defensive frontend filter:** chat UI no longer renders strings starting with `broadcast:` as a sender label if they ever slip through.

### v0.6.3 (April 16, 2026)

- **TX channel hash fix:** messages sent from the dashboard were going out with hash 0x02 (invisible to the mesh) instead of the correct 0x08. The primary channel name defaulted to blank, producing the wrong hash. Now defaults to "LongFast" matching Meshtastic firmware. ([#21](https://github.com/KMX415/meshpoint/issues/21))
- **Primary channel editable:** channel 0 can now be renamed and saved from the Radio settings page. Previously edits reverted on refresh. ([#13](https://github.com/KMX415/meshpoint/issues/13))
- **Channel display cleanup:** Radio settings shows the actual channel name (e.g. "LongFast") instead of "Primary (LongFast)".

### v0.6.2 (April 16, 2026)

- **MQTT channel name fix:** MQTT topics now use the actual channel name (LongFast, MediumFast, ShortFast, etc.) instead of `chXX` hashes. New `ChannelResolver` maps all 8 standard Meshtastic presets and supports user-configured channel keys. ([#20](https://github.com/KMX415/meshpoint/issues/20))
- **Chat sender names:** received messages now show the sender's node name or hex ID. Previously there was no way to tell who sent what. ([#19](https://github.com/KMX415/meshpoint/issues/19))
- **Chat day dividers:** messages from different days are separated by date labels (Today, Yesterday, or the date) in the chat window.
- **Espressif USB udev rule:** installer adds a udev rule so Heltec V3/V4 and T-Beam ESP32-S3 USB serial devices are accessible to the meshpoint service user without manual group changes. ([#12](https://github.com/KMX415/meshpoint/issues/12))

### v0.6.1 (April 11, 2026)

- **Local stats dashboard:** new Stats tab on the local dashboard with 12 live Chart.js charts: protocol split, packet types, RSSI distribution, signal quality, direct vs relayed, active nodes, device roles, hardware models, relay decisions, rejection reasons, and traffic timeline. All generated locally, no cloud needed.
- **Enriched heartbeat:** edge accumulates per-packet stats in memory and sends a batched summary to Meshradar in each heartbeat instead of the cloud processing every individual packet. Same data, significantly fewer backend operations. Savings scale with fleet size.
- **Local topology layer:** map tab gains a "Topology Links" toggle showing lines between nodes with RSSI/SNR tooltips.
- **Farthest direct tracking:** tracks the farthest direct (0-hop) node heard, with distance and signal strength, visible on the stats page.
- **Relay rejection tracking:** relay engine now records why packets are rejected (duplicate, rate limited, type filtered, signal bounds), visible in local stats.

### v0.5.4 (March 30, 2026)

- **MQTT gateway:** dual-protocol MQTT publishing for Meshtastic (protobuf ServiceEnvelope) and MeshCore (JSON). Publishes to community maps (meshmap.net, NHmesh.live) and Home Assistant. Two-gate privacy model: MQTT is off by default, and only public channel traffic is published unless you explicitly allowlist a private channel. Each Meshpoint gets a unique node-format gateway ID that integrates natively with the Meshtastic ecosystem, appearing on meshmap.net, Liam Cottle's map, and other community tools. Optional JSON mirror for HA/Node-RED, auto-discovery sensor configs, and configurable location precision.
- **Packet type filter (cloud):** filter the Meshradar cloud packet feed by type (traceroute, position, text, etc.) and protocol (Meshtastic/MeshCore). Dropdown filters in the packets tab header.
- **Setup wizard MQTT step:** `meshpoint setup` now includes an MQTT opt-in prompt with broker selection and HA integration toggle.
