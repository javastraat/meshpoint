# Changelog

### Unreleased

- **MQTT broker TLS.** Transport TLS (`mqtts`, CA bundle, cert validation) is not implemented on `mqtt_publisher.py` (plain TCP only). Until then use plain port 1883 or a LAN broker without TLS.

### v0.7.7 (July 2026)

First tagged release of the javastraat/meshpoint fork: LoRaWAN sniffing, multi-radio capture, the RTL-SDR web listener, and a dashboard self-update repair -- plus everything from upstream's own v0.7.7 (backup/restore, RF Environment tab, mesh broadcast cadence, operator tools; merged, bullets below). Run `install.sh` when upgrading: new sudoers rules for backup restore. **Upgrade note:** boxes on v0.7.6 cannot fetch this release from the dashboard (that is the bug being fixed); one manual round on the gateway is required: `cd /opt/meshpoint && sudo git fetch origin main && sudo git reset --hard origin/main && sudo systemctl restart meshpoint`. The restart installs the corrected sudoers rules; Check/Apply works from the dashboard again afterwards.

#### LoRaWAN sniffing (SX1302)

- **Passive LoRaWAN capture on EU868.** Five real 125 kHz LoRaWAN channels (867.9–868.7 MHz, sync word 0x34) on SX1302 ch0–ch4 while Meshtastic stays on the 250 kHz service channel (869.525 MHz, 0x2B) — dual sync word via direct service-channel register writes after `lgw_start()`.
- **LoRaWAN MAC decoder.** Join-Request, Data Up, and Rejoin frames parsed (DevEUI/AppEUI, DevAddr, FCnt, FPort, MIC); payloads stay encrypted (no session keys), listen-only.
- **LoRaWAN dashboard page.** Devices, recent packets, and stats views backed by `GET /api/lorawan/devices`, `/api/lorawan/packets`, `/api/lorawan/stats`.
- **Strict isolation.** LoRaWAN traffic is never relayed and never enters the mesh node roster or telemetry store; router checks LoRaWAN before Meshtastic to avoid false positives.
- **CSV export on every protocol page.** An Export button on the LoRaWAN, Meshtastic, and MeshCore pages downloads the active tab's dataset — all captured packets, or the full device/node/contact census (not just the last 100 on screen). Streamed row-by-row from the database so a full-table packets export never buffers in memory; UTF-8 with a BOM so Excel renders the emoji-heavy MeshCore names; packet exports carry the decoded payload as a JSON cell, and LoRaWAN additionally flattens DevEUI/FCnt/FPort/MIC into their own columns for coverage and frame-counter-gap analysis. Six new `GET /api/{protocol}/export/{packets,census}.csv` endpoints (viewer-open, authenticated by the session cookie).
- **FPort/FCnt show up in the packets log.** The LoRaWAN page's FPort and FCnt columns were permanently "--": the API read the decoded payload with underscored key names (`f_cnt`) while the decoder stores them without (`fcnt`). The frame counter and port now display for every data packet.
- **LoRaWAN packet-detail decrypt state now matches the live dashboard feed.** `GET /api/lorawan/packets` now returns each row's persisted `decrypted` flag, so opening the same packet from the LoRaWAN page and from Dashboard no longer disagrees on "Decrypt: Success" vs "No matching key".
- **LoRaWAN packet detail keeps metadata visible when payload keys are missing.** For encrypted LoRaWAN uplinks without app-session keys, Packet detail now still shows the decoded MAC/header JSON (`dev_addr`, `fcnt`, `fport`, `mic`, etc.) instead of hiding it behind only "No matching key"; modem CR is also suppressed on LoRaWAN rows to keep Dashboard and LoRaWAN-tab modals visually aligned.
- **Clicking a node name inside a Messages thread now opens the same node detail drawer as the dashboard/protocol tabs.** Works for the chat header's contact name (DM threads only — a channel's header is the channel, not a node) and for each received message's sender label in both DMs and channels. DM sender IDs were already reliable (`msg.node_id` is the one contact); channel/broadcast messages needed a small backend addition first — the server already looked up a broadcast message's real sender via `packet_repo.get_source_id_by_packet_id()` just to build its display name (`message_name_resolver.py`'s `_resolve_broadcast_sender`) and then discarded the ID, so the frontend had no reliable node ID to click through to for a channel message. Now stashes it as `source_id` on the returned message dict alongside the existing `node_name`. **Fixed same day, found live**: channel sender names still weren't clickable after that first pass — `_resolve_broadcast_sender()` (the only place setting `source_id`) was only ever called when a message had no already-resolved name, but real channel messages almost always already carry one from when they were stored, so the lookup — and `source_id` — was skipped for the common case. Now always resolves the sender ID (still preferring the stored name for display when one exists).
- **LoRaWAN Devices and Meshtastic Nodes tabs now show a "(N total)" / "(N of M)" count next to their search box**, matching the count MeshCore's Contacts tab already had.
- **Search filter on the MeshCore Contacts tab.** Same `.lw-search` widget as LoRaWAN Devices and Meshtastic Nodes, matching contact ID or name; the "(N total)" count now reads "(N of M)" while a search is active. Rounds out the search rollout across all three protocol pages.
- **Search box was leaking onto the Recent Packets tab on LoRaWAN/Meshtastic.** `.lw-search-wrap` set `display: inline-flex` unconditionally, which silently beats the browser's own `[hidden] { display: none }` rule (author CSS always wins over the user-agent stylesheet, regardless of selector specificity) — so the tab-switch code was correctly toggling `hidden` on the element, but it stayed visibly on screen and unfiltering on the packets view. Added an explicit `.lw-search-wrap[hidden] { display: none; }` override.
- **MeshCore Contacts tab no longer paginates.** Removed the 50-per-page click-through (`_renderNodePage`, the Prev/Next controls) — all fetched contacts now render in one list, same as the Meshtastic Nodes and LoRaWAN Devices tabs, clearing the way for a search filter across the whole contact list next rather than one page at a time. The "(N total)" count next to the tab stays.
- **Search filter on the LoRaWAN Devices tab, now also on Meshtastic Nodes.** A search box (matching device ID/name or packet type) filters the list client-side, with a clear (×) button that appears once you've typed something; the query persists across the 15-second auto-refresh instead of resetting. Piloted on LoRaWAN first, then rolled onto the Meshtastic Nodes tab (matching node ID or name) — same `.lw-search` widget, no new CSS needed since both panels already share `lorawan.css`. MeshCore is next.
- **Service channel RF chain fix.** ch8 now picks the correct radio chain from its frequency instead of hardcoded RF0.

#### Setup wizard

- **Setup wizard lets you skip upstreaming entirely.** The `[4/8] API key` step now asks "Upstream data to meshradar.io?" before prompting for a key; answering no writes `upstream.enabled: false` (no API key required) instead of forcing activation, while still preserving any previously-saved key so re-enabling later doesn't require re-entering it.
- **Setup wizard can set the system hostname to match the device name.** Right after choosing a device name (`[5/8]`), on a fresh install (never on a re-run/upgrade — an existing box already has bookmarks, scripts, and SSH history tied to its hostname) the wizard offers to also set the Linux hostname to match, slugified to a valid form (lowercase, hyphens, 63-char cap) — so the box becomes reachable at `<name>.local` via the mDNS support `scripts/install.sh` now sets up. Runs `sudo hostnamectl set-hostname` and updates `/etc/hosts`' `127.0.1.1` entry to match (hostnamectl alone doesn't touch it, leaving a stale entry that causes "sudo: unable to resolve host" warnings otherwise); skipped automatically if the slug already matches the current hostname, and any failure (e.g. no privilege) is caught and logged without aborting the rest of setup.

#### Multi-radio capture (5 networks at once)

- **Multiple MeshCore USB companions.** `capture.meshcore_usb` accepts a list (up to 4) with per-stick labels; packets carry `meshcore_usb_<label>` as capture source. New `PUT /api/config/capture/meshcore-companions` replaces the list atomically; the Configuration page grew a dynamic companion section.
- **Multiple Meshtastic USB sticks.** `capture.serial` accepts the same list-plus-label shape as `meshcore_usb` (e.g. one stick on 433 MHz, one on 868 MHz), packets tagged `serial_<label>`; the legacy single `serial_port`/`serial_baud` fields keep working unchanged when the list is empty. Also fixes a latent duplicate-capture bug: meshtastic-python publishes every open USB interface's packets on one process-wide pypubsub topic, so running more than one stick without an identity check would have cross-attributed packets between devices. New Configuration → Serial page edits this list from the dashboard (add/remove/label, same shape as the MeshCore companion editor) instead of hand-editing `local.yaml`; new `PUT /api/config/capture/serial-devices`.
- **Serial Meshtastic capture crash and silent packet loss fixed.** The first time a `serial` USB source actually ran, it crashed the pipeline (`TypeError: object of type 'MeshPacket' has no len()`) — meshtastic-python always hands back the raw protobuf object, not bytes, so the old code never fell back to header reconstruction. Fixed, plus two bugs the crash was hiding: the reconstruction fallback read the wrong dict key (`encoded`/hex instead of the real `encrypted`/base64), and it required a `decoded` key that is absent by design on exactly the packets a passive sniffer cares about most — those the connected stick's own key couldn't decrypt (traffic on a channel it isn't configured for). Those were silently dropped instead of being handed to Meshpoint's own decoder/`channel_keys`.
- **Serial Meshtastic packets the connected stick decrypts itself now show their real type.** `encrypted`/`decoded` share one protobuf oneof on MeshPacket, so a packet the stick's own key successfully decrypted locally has no ciphertext left at all — these always showed as "Unknown" even though the decoded content (position, telemetry, nodeinfo, text) was sitting right there in the packet dict, discarded before Meshpoint ever saw it. `RawCapture` gained an optional `pre_decoded` field (portnum + inner payload bytes); `MeshtasticDecoder.decode()` dispatches it directly instead of attempting (and failing) to decrypt an empty body, so these packets now decode to their real `PacketType` just like concentrator-captured ones.
- **Band tag on node cards and the drawer.** Nodes now show a "433 MHz" / "868 MHz" chip derived from their latest packet's labelled capture source (`serial_433`, `meshcore_usb_868`, `concentrator`, …), so a 433/868 multi-stick setup is easy to tell apart at a glance.
- **Real region/frequency/SF/bandwidth/coding-rate for `serial` Meshtastic capture, plus a topbar badge per USB stick.** The `serial` source previously stamped every packet with a hardcoded `906.875 MHz` (a US915 placeholder) and SF11/BW250 regardless of the connected stick's actual region or preset. It now reads the stick's real config at connect time (meshtastic-python completes this handshake automatically before the connection call returns) — region, channel, primary channel name, modem preset (or the raw custom SF/BW/coding-rate when not using a named preset) — the same "read once at connect, mirrors MeshCore's `self_info` fix" pattern. New `src/radio/channel_frequency.py` computes the actual operating frequency using the same formula as `meshtastic/firmware`'s `RadioInterface.cpp` (verified against a local firmware checkout and cross-checked against a real device's observed frequency): region band, modem bandwidth, and — for the default/auto channel — a djb2 hash of the primary channel name (or the modem preset's display string when the channel name is blank), exactly reproducing the firmware's own frequency-slot selection instead of a fixed per-region guess. An explicit non-default channel, or one outside the region's valid slot range, reports `0.0` (this codebase's existing "unknown" sentinel) rather than a wrong number. New topbar badge — one small cyan chip per configured Meshtastic USB stick, laid out the same as the Meshtastic and MeshCore chips (brand, lamp, the stick's own node ID as the "call sign" slot, region, frequency, and its modem preset) — separate from the single "primary" MeshCore companion chip, since every serial device is capture-only (no TX binding to pick "the" one); reacts to a dashboard websocket drop the same way the other two chips do (amber "reconnecting" lamp, region/frequency/preset blanked) instead of silently keeping stale data on screen through a restart. New `scripts/backfill_serial_meshtastic_signal.py` corrects old rows still carrying the `906.875` placeholder.
- **Serial Meshtastic sticks no longer capture their own self-telemetry as spammy, confusing packets.** A connected stick's own periodic self-report (battery/uptime NodeInfo/Telemetry) passes through the same event stream as genuinely received packets — meshtastic-python doesn't distinguish them — and firmware's own convention (`rx_rssi == rx_snr == 0` meaning "generated locally, never actually received over the air") confirmed these had no real signal to show; they were landing in the packet feed and node counters as a confusing `-100 dBm` reading of the stick reporting on itself, roughly once a minute. The connect-time handshake now also reads the stick's own node number (`interface.myInfo.my_node_num`, no config needed) and drops any packet it originates before decode/storage/the feed/counters ever see it. The detected ID is exposed via the API and shown in the topbar badge's tooltip for visibility. New `scripts/purge_self_originated_node.py` removes historical packet/telemetry/message/node rows left over from before this fix (the drop only stops future capture, it doesn't retroactively clean the database). The drop now exempts `TEXT_MESSAGE_APP`: a chat message typed via a BLE/WiFi-connected app on the same physical stick is genuine content the user wants in the Messages panel, not a beacon, so it still reaches decode/storage even though the stick is its own "from" node for that packet — only routine self-reported telemetry/nodeinfo/position stays dropped.
- **Labelling a MeshCore companion no longer silently disables its TX/advert/status.** The dashboard's "MeshCore connected" lookup matched capture sources by exact name (`"meshcore_usb"`), which breaks the moment a companion gets a label (its name becomes `meshcore_usb_<label>`) — no crash, just a topbar stuck on "No companion" and no startup advert. Fixed to the same `startswith` rule already used for packet routing. Receive/capture was never affected, only transmit and the live-status chip.
- **MeshCore signal metadata.** Frequency/bandwidth/SF stamped from each companion's connect-time radio handshake (per stick, so 868 and 433 report their real channel), and hop count derived from `path_len` on contact/channel messages.
- **Simultaneous LoRaWAN + Meshtastic 868 + MeshCore 868/433 + Meshtastic 433.** Concentrator, two labelled MeshCore sticks, and a Meshtastic serial node run side by side; only Meshtastic is ever relayed.
- **DM replies now go out through the same radio that can actually reach the contact, not always "the" primary one.** On a multi-radio box (2+ MeshCore companions, or a concentrator plus a 433 MHz Meshtastic USB stick), every outbound reply previously routed through one fixed radio regardless of which one had actually heard the contact — confirmed live ("when sending thru a remote 433 companion i receive the message in chat... but when i reply... it never arrives"): a contact heard only on 433 MHz physically cannot receive a reply sent out on 868. New `NodeRepository.get_latest_capture_source()` looks up which capture source most recently heard a given contact; `TxService` uses it to pick the matching companion/stick and call its own send method instead of the primary's, falling back to the old fixed-primary behavior whenever there's no packet history yet or the matched source is disconnected. MeshCore channel broadcasts now fan out to every connected companion (channel keys are mesh-wide, so a broadcast should reach every mesh a companion is bridging) rather than going out on just one. Meshtastic serial USB sticks gained message-send capability for the first time (`SerialCaptureSource.send_text()`, previously admin-only: nodeinfo/owner); Meshtastic broadcasts still go out on the concentrator only, unaffected by this fix.
- **Chat header shows which physical connection a conversation is on.** A small pill next to the existing MC/MT protocol badge names the specific companion/stick (e.g. "433", "868", "Concentrator") a contact was last heard through — useful once a box has more than one radio per protocol and the protocol badge alone can't tell them apart. Backed by the same `get_latest_capture_source()` lookup as the reply-routing fix above; hidden for brand-new conversations with no packet history yet and for broadcast/channel threads (not tied to one specific connection). **Fixed same day, found live**: the pill didn't render at all on a real deployment even though the backend data was correct (verified against a real DB copy — `capture_source` resolved correctly for both a 433 and an 868 contact) — root cause was `.msg-chat__connection-badge` never getting any CSS at all, unlike its sibling `.msg-chat__protocol-badge`, so it had no background/padding/color and was effectively invisible next to the styled MC/MT badge. Added matching pill styling in `frontend/css/messaging.css`. **A second, more serious bug found the same day (real live report: "no freq shown... also it takes at least 15 sec before chat is loaded")**: `_enrich_conversations()` was awaiting the capture-source lookup once *per conversation*, sequentially — every part of the app shares one `aiosqlite` connection that funnels all DB access (including live packet writes from every capture source) through a single background-thread queue, so a dozen-plus conversations each queued up behind ongoing capture writes, visibly slowing the whole Messages page. New `NodeRepository.get_latest_capture_sources()` (batched, one query for every conversation's node_id via the same `ROW_NUMBER() OVER (PARTITION BY source_id ...)` pattern already used elsewhere) replaces the N sequential awaits with one. **The Messages page was still slow after that fix**, tracked down to a pre-existing (not new this session) bottleneck: `MessageNameResolver._lookup_meshcore()` calls `get_contacts()` to resolve a MeshCore contact's display name, and that call is a live USB round trip to the physical companion (up to a 10 s timeout) — invoked once per conversation and once per message on a 50-message chat page, so a single page load could trigger dozens of real hardware round trips back to back. Added a 10-second cache so a page load reuses one fetch instead of one per item. **Channel/broadcast conversations now get the pill too** (previously DM-only): since packets aren't tagged with a channel number, this is scoped to "which capture source most recently heard this protocol" rather than the exact channel — coarser than the per-contact DM lookup, but still tells 868 and 433 apart, or shows "Concentrator" for Meshtastic. New `NodeRepository.get_latest_capture_sources_by_protocol()`. That backend addition alone wasn't reachable from the UI, though: the sidebar's channel rows are built client-side from `/api/messages/channels` (pure config — names/keys, no capture data) by `_channelToConvo()`, which merges in a few fields from the enriched conversations list but didn't copy `capture_source`, so a channel's pill could never render no matter what the API returned — fixed. The whole backend chain was also proven end-to-end by running the real route code against a production database copy: every DM and channel conversation resolved its correct source (DM on the 433 companion → `meshcore_usb_433`, MeshCore channels → `meshcore_usb_868`, Meshtastic channels → `concentrator`) — and then confirmed on the live deployment itself (the user pasted the real `/api/messages/conversations` response, every row carrying the right `capture_source`). **The pill now also shows in the conversation list rows** (next to each channel/DM's MC/MT badge in the sidebar), not just the open chat's header — the list is where a multi-radio setup actually scans for "which band is this on", and it was the spot the "no freq shown on channels and dm's" reports were about all along; the full source name appears as a hover tooltip. **Live-verified**, which immediately surfaced one layout follow-up: a long node name (e.g. "PD2EMC Companion 🏠") pushed the badges out of the row entirely — the name row is a flex container, so its `text-overflow: ellipsis` never actually applied to the text and long names just crushed the badge elements instead. The name now truncates with "…" in its own span while the MC/MT and connection badges always stay visible.
- **Messages page headers were permanently invisible — the panel was exactly one topbar too tall.** The chat header (contact name, protocol badge, connection pill) and the sidebar's own "Messages / + New" header had never been visible on a real deployment ("there is nothing above 2day my friend"): `.messaging` sized itself `calc(100vh - 32px)` inside a `100vh` overflow-hidden container that also holds the ~48px topbar, and opening any conversation focuses the compose input — which auto-scrolls the clipped container to reveal it, shoving both headers up under the topbar with no scrollbar to ever bring them back. The panel now sizes from its flex parent (which already accounts for the topbar) instead of viewport math, on desktop and mobile both; the compose bar and both headers now fit on screen simultaneously.
- **Channel pills are now per-channel, not protocol-wide.** With two same-protocol companions, the first implementation made every MeshCore channel's pill flip in unison to whichever companion heard the latest MeshCore packet anywhere — all channels suddenly read "433" ("we cant find our channels back"). Each stored message carries the packet_id it came from, so a channel's pill now comes from joining its latest received message back to that packet's actual capture source (`NodeRepository.get_capture_sources_for_broadcasts()`); the protocol-wide lookup remains only as a fallback for channels with no received messages yet. New `scripts/backfill_meshcore_capture_labels.py` (dry-run by default) relabels pre-fix bare `meshcore_usb` packet rows using each companion's own frequency band, learned from the already-labeled rows rather than hardcoded — verified on a production DB copy (2142 rows relabeled, every channel pill then showing its true band).
- **Topbar chip row didn't wrap on narrow (mobile) screens.** A second MeshCore companion (or Meshtastic USB stick) chip could run off the edge of the screen and never be visible at all — `.topbar__group`, the wrapper around same-protocol chips, was `inline-flex` with no `flex-wrap`, so chips that didn't fit on one line overflowed instead of dropping to a new line. Added `flex-wrap: wrap`.
- **Concentrator Meshtastic TX was RF-invisible to every Meshtastic radio — wrong LoRa sync word on transmit.** Reception has always worked (the service-channel demodulator is programmed to Meshtastic's 0x2B sync word at startup), but that fix only touches RX registers: the TX modulator is programmed per-send inside the stock HAL's `lgw_send()`, which only knows LoRaWAN public 0x34 (our board config, needed for LoRaWAN reception on the multi-SF channels) or private 0x12 — 0x2B does not exist on the TX side at all. So every dashboard message, nodeinfo, ACK, or telemetry sent via the concentrator was transmitted with the LoRaWAN sync word and silently discarded by every Meshtastic radio's demodulator; "sent" in the UI, nothing on the mesh. Found live-debugging "tag to Meshpoint works, Meshpoint to tag never arrives" after everything else (channel hash, PSK, frequency, SF/BW, AES) checked out as correct in the TX logs. An earlier `patch_hal.sh` had already tried to fix this by snapshotting TX peaks inside `sx1302_lora_syncword()` — but that function is only ever called by `lgw_start()` with the board's `lorawan_public` flag, so once `lorawan_public=True` became the default for LoRaWAN capture, the snapshot was always 0x34 and the patch silently did nothing (the RX 0x2B fix bypasses that function entirely with direct register writes). `patch_hal.sh` now injects an explicit exported `sx1302_set_tx_syncword()` override applied to SF7-SF12 transmissions (SF5/SF6 keep the fixed 0x12 their fine-sync path requires), restoring the stock source from git first when the old patch is present; the Python wrapper sets it to the same sync word as RX at startup, and `post_update.sh` re-patches automatically on the next update (or run `sudo /opt/meshpoint/scripts/patch_hal.sh` manually). On a `libloragw.so` built without the new symbol the service logs a loud warning that concentrator TX stays Meshtastic-deaf instead of crashing. The vendored reference copy (`extra/sx1302_hal`) carries the same patch.
- **CI test suite fixed after a real regression shipped unnoticed.** GitHub Actions caught 11 failures across the companion-name reapply, rename route, and serial-enrichment tests — earlier per-companion refactoring this release (rename/reapply, per-entry companion names) had changed function signatures and where names get persisted, but the corresponding `tests/` files were only ever checked with ad-hoc stub scripts, not the real pytest suite CI runs. Test fixtures updated to match the new `_reapply_companion_name(source, config, is_primary)` signature and the new per-entry `capture.meshcore_usb[i].companion_name` persistence location; full suite now verified locally (1166 passed) before shipping, closing the gap that let this slip through.

#### RTL-SDR web listener (Radio tab)

- **Browser radio via RTL-SDR dongle.** `rtl_fm` → ffmpeg MP3 streaming with WFM/NFM/AM/USB/LSB modes, tune-anywhere, squelch/gain/level controls, 10-minute idle auto-stop; `GET/POST /api/listener/*` endpoints behind the session cookie.
- **RDS on broadcast FM.** Wideband MPX pipeline teed to redsea: station name, RadioText, programme type, and a BLER signal-quality pill; EU 50 µs de-emphasis reconstruction.
- **Two radio faces.** Digital skin (VFD-style frequency readout, segmented VU) and Analogue skin (slide-rule dial with band scales, preset flags, swinging-needle VU); choice persists per browser.
- **Preset stations.** Favorites with pinning, category tabs, and search across Amsterdam FM, PMR446, marine VHF/UHF, Schiphol airband (true 8.33 kHz carriers), and ham presets; now-playing markers on tab and row. Preset display switched from a wrapped chip grid to a list of rows (star, name, frequency) matching the DAB+ tab's station-list look, once that tab established the pattern -- same category tabs and click-to-tune behavior, just restyled; the frequency is now visible directly on each row instead of only appearing after tuning.
- **Real-time VU meter.** Client-side Web Audio analyser (instantaneous RMS at ~60 fps with peak hold) instead of server-side loudness polling.
- **Clean retunes.** Stop → settle → start pipeline with retry fixes the "Failed to open rtlsdr device #0" race on fast channel switches.
- **P2000 and Pagers tabs.** Two new tabs alongside Radio, decoding live pager traffic instead of streaming audio: P2000 (Dutch emergency dispatch, FLEX on 169.65 MHz) and Pagers (POCSAG512/1200/2400 on 172.45 MHz), each a fixed `rtl_fm | multimon-ng` pipeline with its own Start/Stop button and a live-scrolling decoded message log (timestamp, protocol, capcode, message text). Since the RTL-SDR dongle can only be tuned to one frequency by one process at a time, only one of Radio/P2000/Pagers can be active at once — starting one while another is running returns a clear error rather than silently stopping it (stop the active one first), and all three tabs now show "busy — in use by X" (with Start/Tune disabled) whenever a sibling tab holds the dongle, instead of a bare "idle" that looked like nothing was using it. New `src/audio/pager_listener.py`, `src/audio/sdr_registry.py` (the shared "who owns the dongle" check), and `GET/POST /api/p2000/*` + `/api/pagers/*` endpoints. **P2000 (FLEX) fixed 2026-07-13**: real FLEX pages were being silently dropped — `_FLEX_RE` was written against a colon/space format guessed from documentation (no RTL-SDR hardware to test against at build time), but real multimon-ng output is pipe-delimited (`FLEX|<timestamp>|<baud>/<level>/<phase>/<cycle>|<frame>|<capcodes>|<kind>|<message>`), confirmed by running the exact pipeline manually in a shell and capturing a real page. The old regex's "recognized protocol, unmatched format, show raw anyway" fallback also only checked for a `FLEX:` prefix, so real pipe-delimited pages fell all the way through to being ignored like a blank line, not even shown raw. Rewritten to match the real format; when a page lists multiple space-separated capcodes (simulcast/alternate addressing) the first is shown in the compact capcode column, with the full line always preserved in `raw`.
- **POCSAG tab.** A fourth tab covering the POCSAG512/1200/2400 decoders on a separate 439.9875 MHz band from the Pagers tab, sharing the same `rtl_fm | multimon-ng` pipeline shape and dongle-exclusivity rules. The message parser was confirmed against real captured multimon-ng output on the Pi (`POCSAG1200: Address: 2041152  Function: 3  Alpha:   Test message<NUL><NUL>`), which also revealed that multimon-ng pads POCSAG alpha text with literal `<NUL>` tokens — now stripped for a clean message display. New `GET/POST /api/pocsag/*` endpoints.
- **Digital/Analogue skin toggle and status pill moved onto the Radio tab's own card.** They previously lived in the page-level header, which meant they kept showing on the P2000/Pagers/POCSAG tabs too even though those tabs have no skin or "on air" concept of their own — cosmetic leftover from before those tabs existed. Fixed a follow-on side effect of that move: the status text (e.g. "busy — in use by P2000") inherited the card header's uppercase/bold styling and looked shouty compared to the same text on the P2000/Pagers/POCSAG tabs — now normal case and weight everywhere. The status dot is also now red while busy (a sibling tab holds the dongle) instead of the same grey as plain idle, keeping green reserved for "this tab is actually on air" — applied consistently across Radio and all three pager tabs.
- **Sidebar shows when the RTL-SDR dongle is in use, from any page.** A green-dot badge on the RTL-SDR nav item names which tab currently holds it (Radio/P2000/Pagers/POCSAG/RTL433), so an operator elsewhere in the dashboard doesn't have to open the Listener page just to check. Polls the existing shared `dongle_owner` status field (same one that powers the "busy — in use by X" text within the Listener page itself) every 5 s; hidden when the dongle is free. New `frontend/sidebar/listener_badge.js`, mirroring the existing NodeInfo TX countdown badge's pattern. The graph builds at request time from stored packets (`GET /api/topology/graph`) and fills in on its own as traceroutes and neighbour imports accumulate. With repeater polling enabled, the graph also stays **live**: each poll now asks the repeater for its own neighbour list (`fetch_all_neighbours`, same logged-in session as status/telemetry, tolerant of firmware that lacks it), every polled repeater becomes its own star of SNR edges with skew-immune freshness timestamps, and the Repeaters page health card shows the fetched neighbour count. The Signal Intelligence section pairs the RSSI histogram with an SNR histogram (2 dB buckets over the last 500 packets), so noise-margin health is visible at a glance. An **Export PNG** button (Graph mode only) snapshots the current view — node/edge counts and a timestamp are stamped onto the image itself so it's self-contained outside the dashboard; Map mode is a separate Leaflet layer and isn't covered by this button yet.
- **Recent packets in the node drawer.** Opening a node now shows its last 15 packets (time, type, RSSI/SNR to one decimal) in a collapsible section, next to the existing metrics history; works for Meshtastic and MeshCore nodes alike.
- **Near-field packets no longer skew signal stats.** RSSI readings above −20 dBm (a node centimetres from the antenna during desk testing, or 0-dBm placeholders) are excluded from the Stats best/avg RSSI tiles and the RSSI histogram, so "Best RSSI" reflects actual reach.
- **Network totals no longer capped at 500 nodes.** The network summary (Stats page and `meshpoint report`) computed its totals over the 500 most recent nodes, silently under-reporting node counts, positions, and packet sums on larger installs; it now uses whole-table SQL aggregates.
- **Total Packets tile shows 24h / total.** The dashboard stat card now reads like the Nodes Discovered tile ("339 / 935"), pairing the last 24 hours with the all-time count.
- **"Farthest Direct Signal" no longer shows an imported repeater.** The fallback query behind this stat (used when no genuinely-captured direct reception exists yet) picked the max-distance `neighbour_advert` row with no regard for where the row came from — including historical MeshCore neighbour imports, which insert third-party observations with `hop_limit=0` (looks direct). A freshly-imported repeater 42 km away could outrank every packet this box has actually heard itself. The query now excludes both known import markers: `capture_source='meshcore_db_import'` (current `scripts/import_meshcore_db.py`) and the `nb:`/`meshcoredb:` `packet_id` prefixes (the actual marker on already-imported historical rows, from an earlier now-superseded import script) — genuine direct receptions never use either pattern.
- **Range card subtitles on the Stats page now say what they actually measure.** "Farthest Direct Signal" was labeled just "Received directly without relaying (0 hops)" — accurate but easy to misread as band- or protocol-specific; it's actually the single farthest zero-hop reception across every protocol/band this box listens to (Meshtastic or MeshCore only — LoRaWAN packets never get a position profile, so they can never appear here), tracked live for the current run. "Farthest MeshCore Contact" was the more misleading of the three: despite sitting next to two reception-based cards, it's not a reception measurement at all — it's the farthest node in your MeshCore roster with a known position, which can include imported or historical contacts you've never actually heard directly. Both subtitles now say so.
- **"Farthest Direct Signal" is now purely session-based, no DB fallback.** User noticed a node the Dashboard's Nodes panel correctly tagged DIRECT never showed up as "Farthest Direct Signal" on the Stats page — traced to the stat's DB-backed fallback (`_get_farthest_neighbour_direct()`, used when the live session accumulator has nothing yet) being hardcoded to `packet_type = 'neighbour_advert'`, MeshCore-only, so a Meshtastic direct contact could never satisfy it regardless of distance or recency. First fix broadened the fallback to match by hop count (`hop_start = 0` or `hop_start = hop_limit`, the same definition the Dashboard's own DIRECT filter uses) instead of packet type — but this surfaced a physically implausible 743 km "direct, no relay" MeshCore contact with a strong clean SNR, exposing a deeper problem: `hop_start`/`hop_limit` default to `0`/`0` on several packet paths that are *not* genuine direct receptions (confirmed live repeater-neighbour-poll rows default to the identical 0/0 and were only ever excluded from the old fallback by a coincidental `nb:`-prefixed packet_id convention, not by anything hop-related) — hop count alone isn't a trustworthy signal of "really heard directly" across every row in the table. Removed the DB fallback entirely rather than keep patching it; "Farthest Direct Signal" now only ever reflects `StatsReporter`'s live, real-time hop accounting for packets actually decoded this session — it can legitimately read "--" right after a restart until something genuinely direct is heard again, which is more honest than a DB query second-guessing historical hop metadata.

#### Configuration and server

- **Named, revocable API keys for Meshpoint's read-only status routes.** `require_auth` on `/metrics` previously only accepted the dashboard's own session cookie or a short-lived session Bearer JWT, which is awkward for an unattended scraper (Home Assistant, Prometheus) that can't hold a browser login. Configuration → Metrics now has an "API keys" panel: generate a labelled key (e.g. "Home Assistant"), copy the raw value shown once (only its SHA-256 hash is ever stored, matching how admin passwords are handled), and send it as `Authorization: Bearer <key>`. Each key is scoped to a small fixed allowlist — `/metrics`, plus (new) `GET /api/device/metrics` (host CPU/RAM/disk/temp/fan) and `GET /api/stats/summary` (best signal ever, farthest contact, node role/hardware-model distribution) — never general dashboard access, config mutation, or mesh control. Individually revocable, takes effect immediately with no restart. New `MetricsApiKey` config model (`metrics.api_keys`), `POST`/`DELETE /api/config/metrics/api-keys`, shared `src/api/auth/metrics_api_key.py` dependency used by all three routes.
- **Fresh installs no longer crash `GET /api/config`.** On a new box whose `local.yaml` has a bare `meshcore.private_channels:` (or `channel_keys:`) key, YAML parses the value as null, overwriting the config's empty-list default — every dashboard page then hit a 500 (`'NoneType' object is not iterable`) when loading configuration. Both fields are now null-guarded in the endpoint.
- **Installer ships the lgpio build dependencies.** `pip install lgpio` (fan/LED/button support) compiles a C extension that needs `swig` + `python3-dev` to build and `liblgpio-dev` to link — previously documented in TROUBLESHOOTING.md but missing from `scripts/install.sh`, so fresh installs and upgrades died at "Setting up Python virtual environment" with `error: command 'swig' failed`. The installer now apt-installs all three, both in the main package list and before the upgrade fast-path's early pip refresh — and also ships `ffmpeg`, which the RTL-SDR listener's audio pipeline needs.
- **Installer sets up RTL-SDR support end to end.** Previously entirely manual per box: `librtlsdr` wasn't installed at all, the dongle had no udev permissions, and the kernel's own DVB driver stack (`dvb_usb_rtl28xxu` plus the `rtl_2832`/`rtl_2830` demodulator modules it depends on) would claim the RTL2832U chip before rtl-sdr's userspace tools (`rtl_fm`/`rtl_test`) could open it. The installer now blacklists all three kernel modules (and unloads them immediately if already loaded this boot, bridge module first since it depends on the demodulator modules, so a reboot isn't required to test), then clones and builds `rtl-sdr` from source (osmocom upstream, `cmake -DINSTALL_UDEV_RULES=ON`) rather than `apt install rtl-sdr` — matching the existing SX1302 HAL build's build-from-source pattern, and leaving room to swap in a vendor fork later (e.g. RTL-SDR Blog's own fork for their V4/R828D dongle) without an apt package fighting it. Both steps are idempotent (skips the blacklist file if already present, skips the clone+build if `librtlsdr` is already installed) — safe to re-run on upgrade. The `meshpoint` service user's `plugdev` group membership (needed for device access) was already handled elsewhere in the installer.
- **Installer builds redsea, the RTL-SDR RDS decoder.** Adds a `scripts/install.sh` step (right after the rtl-sdr build) that apt-installs `meson`/`libsndfile1-dev`/`libliquid-dev` and then clones and builds `redsea` (windytan/redsea upstream) via `meson setup build && meson compile && meson install`, giving the Radio tab's FM listener access to station name, radio text, and PI code decoding on top of the librtlsdr built in the previous step. Idempotent — skips the clone+build if a `redsea` binary is already on `PATH`.
- **Installer sets up mDNS (Avahi) so the box is reachable as `meshpoint.local`.** Previously an IP address was the only way to reach a fresh box — awkward right after flashing, before a static IP/DHCP reservation exists, and for the Home Assistant integration's setup step, which asks for a host. `apt install avahi-daemon avahi-utils` + `systemctl enable --now avahi-daemon`, idempotent (skips the apt install if `avahi-daemon` is already present).
- **Unmapped Meshtastic channel hashes no longer silently blend into LongFast's message history — and no longer vanish from the Messages sidebar either.** `ChannelHashResolver.lookup()` used to fall back to channel index 0 for any `channel_hash` it couldn't map to a configured channel — meaning traffic on a channel this box doesn't have configured (a config mismatch, e.g. the same PSK reused under a different channel name on another node) looked identical to genuine LongFast messages, with nothing distinguishing them apart. Confirmed this masked a real mismatch for weeks in one case. `lookup()` now returns `None` for an unmapped hash instead of `0`, and the caller (`on_text_packet` in `server.py`) routes it to its own distinct conversation — `broadcast:meshtastic:unmapped:0xHH`. Live-testing the backend fix surfaced a second bug in the same feature: the Messages sidebar's "Channels" section rendered exclusively from the configured `/api/messages/channels` list, and "Direct Messages" explicitly excludes anything broadcast — so a correctly-routed unmapped conversation had nowhere to render at all, silently invisible despite being correctly stored and correctly kept out of LongFast. Added a third sidebar section, "Unmapped," in `messaging_contacts.js` specifically for broadcast conversations that exist in stored messages but aren't in the configured channel list. A warning still logs once per unique unmapped hash (existing throttle, unchanged) to avoid flooding logs on a channel with ongoing unmapped traffic. Live-testing also caught a third bug: replying from an Unmapped conversation used `convo.channel || 0` in `messaging.js`, and no `Conversation` object actually has a `channel` field, so every reply silently transmitted on channel 0 (LongFast) instead of failing — the exact silent-wrong-channel bug this feature was built to eliminate on receive, reappearing on send. Fixed by disabling the input/Send button entirely whenever a conversation is genuinely unmapped.
- **Meshpoint can now reply correctly to a channel whose PSK it has configured under a different local name, instead of just refusing to send.** An "unmapped" hash almost always means the remote node uses the same shared key under a channel name typed differently on their side (e.g. their "PD2EMC" is this box's "Home") — the packet decrypts fine, it just fails the exact-hash lookup. `meshtastic_decoder.py`'s existing key brute-force loop already tried every configured key and knew which one worked, but discarded that fact once decryption succeeded; it's now recorded on the `Packet` as `matched_channel_index`. When a hash is unmapped but a key still matched, `on_text_packet` routes it to a distinct "keyed" conversation (`broadcast:meshtastic:keyed:{index}:0x{hash}`) encoding both which local key/channel to reply with and the exact hash byte the sender expects — recomputing our own hash from our own channel name would produce a value their radio doesn't recognize, since Meshtastic's channel_hash is a lossy one-way digest of name XOR key, not reversible back to a name. `tx_service.send_text()` gained an `echo_hash` parameter that stamps the outgoing packet with this exact byte while still encrypting with the correct key; the messaging UI now shows these as a "Different Channel Name" sidebar section with replying enabled, instead of the disabled "Unmapped" state from the previous fix. **Live-tested and confirmed**: a reply sent from this conversation arrived correctly in the Tag's own channel thread, not LongFast. A live follow-up test then caught a fourth bug in the same conversation: after the reply sent successfully, the thread's title regressed from the sender's own name ("PD2EMC Meshtastic Tag") to the raw node_id string. Root cause: `MessageRepository.get_conversations()` grouped by `node_id` with a bare (non-aggregated) `node_name` column, so SQLite could pick that value from whichever row happened to be chronologically latest — a sent reply correctly saves `node_name=""` (there's no "sender" to name for something we sent), so the moment that reply became the newest message, the whole conversation's name reverted to blank even though a real name had been resolved moments earlier. This was never reachable before today, since these conversations couldn't be replied to at all until the previous fix landed. Fixed with a correlated subquery that picks the latest **non-blank** `node_name` per conversation independent of which row supplies the preview text/timestamp.
- **Fixed the 433 MHz Meshtastic USB stick misreading its own local channel-table index as a real over-the-air channel_hash, both receiving and sending.** meshtastic-python reports `packet["channel"]` as the connected stick's own channel-table slot (0-7) for anything it decoded locally with its own key — not the real on-air hash — but `serial_source.py` stuffed that value straight into the raw frame's hash-byte position, so the decoder read a small index as if it were a genuine hash and could misroute the conversation (or, worse, silently collide with an unrelated real channel whose hash happened to match a small number). Fixed by reading the stick's own channel table at connect time (index → name, falling back to the modem preset's display name for a blank primary channel, same convention firmware itself uses) and threading the resolved NAME through instead: `Packet` gained a `remote_channel_name` field, and `on_text_packet` now routes by that name first — landing in the matching Meshpoint channel's normal conversation if the names line up, or a clearly labeled `broadcast:meshtastic:named:{name}` bucket if they don't — rather than ever trusting the bare index as a hash. Sending had the mirror-image bug: replying via a serial stick passed Meshpoint's own channel index straight through as if it were that stick's index, though the two are separate physical nodes with independently-numbered channel tables. `tx_service.py` now translates by name (`resolve_channel_index()`) and refuses to send — rather than guessing — when the connected stick has no channel configured under that name at all.
- **Hardened `ChannelHashResolver.rebuild()` against a config/crypto ordering drift that already caused a real production mismatch (the inbound hash map and outbound TX computation disagreeing about which hash belonged to which channel index — 0xC9 vs 0x71 for the same config).** It previously paired the *i*-th channel name from the config's `channel_keys` dict with the *i*-th key from `crypto.get_all_keys()` — two independently-populated collections (`coordinator.py`'s startup loader vs `config_routes.py`'s live-update handlers) that only produce the right hash if both happen to iterate in the same order. Now looks up each channel's key by NAME directly in `crypto._keys`, immune to either collection's ordering. Also had to make sure this didn't introduce a different bug: `crypto._keys` is shared storage for both Meshtastic *and* MeshCore channel keys, so the lookup stays scoped to names actually present in the Meshtastic `channel_keys` dict passed in, rather than hashing in every key crypto happens to hold.
- **Installer builds multimon-ng, the RTL-SDR digital mode decoder.** Adds a `scripts/install.sh` step for POCSAG/AFSK/DTMF/etc decoding out of `rtl_fm`-piped audio. Built from source (EliasOenal/multimon-ng upstream) via CMake — the project dropped its old qt4-qmake build system entirely, so no Qt4 packages are pulled in (and `qt4-qmake` isn't even packaged in current Raspberry Pi OS repos anymore, so the classic qmake instructions floating around online would hard-fail a fresh install). Apt-installs `libpulse-dev`/`libx11-dev` for live audio input and the X11 scope display. Idempotent — skips the clone+build if a `multimon-ng` binary is already on `PATH`.
- **Installer installs rtl_433, a generic RTL-SDR OOK/FSK decoder.** Adds a `scripts/install.sh` step (right after multimon-ng) installing the Raspberry Pi OS `rtl-433` package with `apt install --no-install-recommends` — small (~500 KB) and current enough that a from-source build (like rtl-sdr/redsea/multimon-ng above) wasn't warranted; `--no-install-recommends` skips the optional soapysdr module packages, which nothing here uses. Covers a much broader device set than the P2000/Pagers/POCSAG multimon-ng pipelines — weather stations, TPMS, remote sensors, and hundreds of other 433/315/868 MHz OOK/FSK devices. No dashboard integration yet; this bullet is installer-only.
- **Installer installs the Meshtastic and MeshCore CLI tools.** Adds a `scripts/install.sh` step (right after rtl_433) installing `meshtastic-cli` (`pip3 install --upgrade "meshtastic[cli]" --break-system-packages`, needed since Debian's system Python is PEP 668-protected) and `meshcore-cli` (`pipx install meshcore-cli`, in its own isolated venv — `pipx` is now installed as a system package alongside the other build tools). Neither is used by Meshpoint itself (which talks to radios directly via meshtastic-python/meshcore-python), just handy for admin debugging straight from the Pi's shell.
- **Installer adds a fastfetch login banner.** Appends `fastfetch` to the `pi` user's `~/.bashrc`, so every interactive SSH/console login shows the system-info splash (OS, uptime, CPU temp/load, RAM, interfaces, etc) instead of needing to run it manually. Idempotent — skips if the line is already present.
- **Quieter boot log.** The MeshCore contact sync logged the companion's entire roster — 350 lines per boot on a well-connected stick. It now logs the peer count, the first 10 names, and "… and N more"; the full roster stays browsable on the MeshCore page.
- **Startup banner: one line per capture source.** The banner's single Source/Frequency pair (which only showed the concentrator's Meshtastic channel) is replaced by a per-source list with each radio's real plan: the concentrator's LoRaWAN span + Meshtastic service channel, each MeshCore companion's frequency/SF from its connect handshake, and each Meshtastic USB stick's resolved frequency and region. Sources still handshaking show "(radio info pending)". The logo box's right border also lines up with its corners again.
- **Web server port is now set in the config.** The dashboard bind address comes from `dashboard.host` / `dashboard.port` in the YAML config (override in `local.yaml`, applied on service restart) via the new `src/serve.py` launcher; previously the port was hardcoded in the systemd unit and the config value only affected the startup banner. If the config fails to load, or the configured address cannot be bound (port in use, privileged port, bad host), the server falls back to `0.0.0.0:8080` so the dashboard (and its update/rollback page) stays reachable.
- **Relay burst and RSSI filters editable in Configuration → Transmit.** Burst size and the min/max RSSI relay window (only packets heard inside the window are rebroadcast) can now be set from the dashboard; previously these existed in `local.yaml` and the relay API but had no UI.
- **MeshCore channel limit raised from 7 to 40 user channels.** The configuration card, YAML `meshcore.channel_keys`, and companion slot sync now accept up to 40 user channels (slots 1–40; slot 0 stays Public), up from 7.
- **Unused API endpoints removed.** `GET /api/packets/protocols`, `/api/packets/types`, `/api/nodes/map`, and `/api/telemetry/{id}` (+`/history`) served data already available through `/api/stats/summary`, `/api/nodes`, and `/api/nodes/{id}/metrics_history`; nothing in the dashboard or CLI used them.
- **Prometheus metrics gets a dashboard config page.** `metrics:` was the last config section with no UI at all — enabling the `/metrics` scrape endpoint meant hand-editing `local.yaml`. New **Configuration → Metrics** page toggles `enabled` and `require_auth`; unlike most config pages, both apply immediately (`metrics_routes.py` reads the config live on every request), no restart needed. New `PUT /api/config/metrics`, and `GET /api/config` now includes a `metrics` summary.

#### CLI

- **`meshpoint report` works again.** The CLI report had been broken since API auth landed — every request got 401 and the command misreported "service is not running". It now prompts for dashboard admin credentials and uses a session cookie; `meshpoint status` distinguishes "running, login required" from "unreachable". Running it as root (`sudo meshpoint report`) skips the prompt entirely: with read access to the on-disk signing key the CLI mints its own short-lived admin token — same trust model as `meshpoint reset-password`, no server-side bypass.
- **`meshpoint report` covers all networks.** New PROTOCOLS section (one line per network: packets, nodes/devices, frequency/SF, TX/sniff status) and CAPTURE SOURCES section (SX1302 channel usage, MeshCore companions with serial port and name, last band sweep); the old RADIO CONFIG section slimmed to Meshtastic TX facts, and a MESHCORE section mirrors it (companion state and name, radio settings, TX power, channel list).
- **`meshpoint report`'s CAPTURE SOURCES section now lists Meshtastic USB serial sticks.** Previously only enumerated the concentrator and MeshCore companions — a connected serial Meshtastic radio (e.g. a Heltec V3) produced no line at all, even though `GET /api/config` already shipped its live connection state. Now reads that same `serial` list and prints one line per stick: frequency/SF, connected/disconnected, and the stick's own long name.
- **`meshpoint report` no longer merges two physically separate Meshtastic networks into one stat line.** A concentrator's own Meshtastic channel and a USB serial stick are usually on completely different frequencies (e.g. 868 MHz vs 433 MHz) — different meshes that can never talk to each other — but the PROTOCOLS section's packet/node counts were adding them together under one "Meshtastic" line. `packets.capture_source` was already tracked per-packet, so the fix is query-only: `PacketRepository.get_protocol_distribution_by_source()`/`get_distinct_node_count_by_source()` (new), wired through `/api/analytics/traffic`'s `meshtastic_by_source` and `/api/nodes/summary`'s `meshtastic_nodes_by_source`. The report now prints one "Meshtastic" line per radio once a second source shows up in the packet history, and a new MESHTASTIC SERIAL section (mirroring MESHCORE) lists each stick's radio settings, primary channel, and node ID. Single-radio boxes are unaffected — one source means one merged line, same as before.

#### SenseCap M1 hardware

- **SenseCap M1 onboard LED/button/fan GPIO probe.** No public schematic exists for this board's expansion peripherals; new `scripts/test_gpio_hardware.py` (`gpiozero`, not a runtime dependency) identifies and exercises them. `led`/`button`/`fan` test one pin directly; `button-scan`/`fan-scan` sweep a batch of candidate pins at once (skipping the concentrator's SPI0 bus, the I2C crypto/temp chips, the HAT ID pins, and the concentrator reset lines) rather than guessing pin-by-pin. Confirmed live on real hardware: LED = GPIO 22, button = GPIO 27, fan = GPIO 13.
- **Configuration → Peripherals page.** Edits the SenseCap M1's onboard fan/LED/button settings (`fan:`/`led:`/`button:` in `local.yaml`) from the dashboard instead of hand-editing YAML — one form per peripheral (enable toggle, GPIO pin, and each peripheral's own tuning fields: fan's temp curve/duty/hysteresis, LED's activity-blink flicker, button's hold time and advert cooldown). New `PUT /api/config/hardware/{fan,led,button}`, always flags `restart_required` since all three only start once at app boot. Named "Peripherals" rather than "Hardware" — that name was already taken by the top-level Radio/concentrator page.
- **Temperature-driven PWM fan control.** New opt-in `fan:` config section drives the SenseCap M1's onboard fan from CPU temperature (`/sys/class/thermal/thermal_zone0/temp`) instead of running it flat-out or not at all. GPIO 13 is a hardware-PWM-capable pin on the Pi 4 (BCM2711 PWM1), so speed ramps linearly between `min_temp_c`/`max_temp_c` rather than a blunt on/off relay; a `hysteresis_c` gap and a `min_duty` floor (most small fans stall below some duty rather than spinning slower) avoid chatter and stalling right at the threshold. New `src/hardware/fan_control.py` (`FanCurve` for the pure duty-cycle math, `FanController` for the polling loop), runs as a background task for the app's lifetime same as the noise-floor emitter. Disabled by default — this hardware doesn't exist on RAK V2/Chameleon/DIY builds. `fan: {enabled: true}` alone is sufficient; every other key has a matching default, also documented (with inline comments) in `config/default.yaml` so the full set of knobs is discoverable without reading source. Caught before release: `_apply_yaml`'s `section_map` (the dict `local.yaml`/`default.yaml` sections get merged through) is manually maintained rather than derived from `AppConfig`'s fields, and `fan` was missing from it — the whole section was being silently ignored, so `enabled: true` would have done nothing at all. Fixed, with a regression test asserting `fan:` in `local.yaml` actually reaches `cfg.fan`. New `gpiozero`/`lgpio` runtime dependencies (`requirements.txt`) — `lgpio` specifically because gpiozero's pure-Python fallback pin factory (used when no real GPIO backend is installed) only allows PWM on pins in its hardcoded per-Pi-model table, and rejects a custom carrier board's repurposed GPIO with `PinPWMUnsupported` even though it's a genuine PWM-capable SoC pin; `lgpio` doesn't have that restriction. `FanController` now detects that specific exception and logs the `pip install lgpio` fix directly instead of a raw traceback. Live-verified the full install chain on a real Bookworm/trixie SenseCap M1: `pip install lgpio` builds a C extension with no piwheels wheel available, needing `python3-dev` + `swig` (build tools) and `liblgpio-dev` (the native `liblgpio.so` the extension links against — `cannot find -llgpio` without it) installed via `apt` first; documented as one block in `TROUBLESHOOTING.md`. One more gap surfaced after that: `lgpio` creates its notification pipe directly in `WorkingDirectory` (`/opt/meshpoint`), but `meshpoint.service`'s `ExecStartPre` only ever chowned `config/` and `data/` to the `meshpoint` service user, never the top-level directory — so it still fell back to the PWM-incapable pin factory with `xCreatePipe: ... Operation not permitted`. Fixed in `meshpoint.service` (`ExecStartPre` now hands the whole install tree to the service user recursively — see the self-update bullet below for why recursive), self-healing on every start regardless of how the checkout was deployed. Live-verified working end to end: `Fan control started on GPIO13` with no exception. The duty-change log line stays at `debug` (it was temporarily `info` to verify the ramp live, but with normal temperature jitter it fires on most 10-second polls — thousands of journal lines a day); the dashboard's Fan card is the live view. New stats-bar **Fan** card (after Load Avg) shows the current duty percentage and what it changed from (`"was 35%"`), hidden entirely when fan control isn't enabled; backed by new `fan_duty_percent`/`fan_previous_duty_percent` fields on `GET /api/device/metrics`, sourced from `FanController.current_duty`/`previous_duty` (only updated on an actual change, not every poll, so "previous" reflects the last real transition). Live logs then showed `(was X)` values disagreeing with the previously printed duty (and "changes" logged when nothing visibly changed): the controller compared the freshly computed full-precision duty against the pin's read-back value, which `lgpio` quantizes differently. It now rounds the duty to the 1% steps the log and dashboard display and compares against its own last-set value, so consecutive log lines agree and identical polls stay silent.
- **User button.** New opt-in `button:` config section gives the SenseCap M1's case button (GPIO 27, confirmed via the GPIO probe script) two gestures. Short press: advert on every TX-capable radio — concentrator Meshtastic NodeInfo, MeshCore companion advert, and each Meshtastic USB stick broadcasting its own NodeInfo — serialized 2 s apart because the two 868 MHz signals overlap outright (869.525/BW250 contains 869.618/BW62.5) and same-box TX deafens the neighbouring receiver; a 30 s cooldown stops advert spam, and LoRaWAN stays out by design (pure listener). Hold 3 s: restart the meshpoint service — the recovery that works when the dashboard doesn't; fired detached so it survives its own process dying, sudoers rule already shipped. The status LED narrates every gesture (ack double-blink, cooldown-denied long blink, fast-blink hold warning), and the controller boots disarmed so a button held straight through the restart it triggered can never re-trigger. Poll-based state machine, same testable pattern as the fan and LED. New `src/hardware/button_control.py`; documented in `config/default.yaml` and `docs/CONFIGURATION.md`.
- **Status LED.** New opt-in `led:` config section drives the SenseCap M1's onboard case LED (GPIO 22, confirmed via the GPIO probe script) as a glanceable status light with four states: steady on = service running and every configured capture source healthy; a brief off-flicker per captured packet (a router-style traffic dip — `activity_blink: false` for a calm steady light); 1 Hz blink = one or more capture sources down; dark = service not running (free — the kernel releases the GPIO line when the process dies). Plain on/off GPIO, deliberately no PWM, so it works even without `lgpio`. Health reads the same per-source running flags the capture pipeline maintains, and packet activity polls the existing stats counter at 10 Hz — the capture hot path is untouched. New `src/hardware/led_status.py`, documented in `config/default.yaml` and `docs/CONFIGURATION.md`.

- **Sensors card no longer blanks after a restart.** A poll can succeed on status but return empty telemetry (the companion still settling right after a service restart); that OK-but-empty result used to overwrite the previously stored sensor readings, leaving "No sensor telemetry" until the next full poll cycle. The poller now keeps the last real sensor data whenever a newer poll has none, failed or not.
- **MeshCore repeater monitoring.** New opt-in `repeater_poll:` config section periodically queries MeshCore repeaters you operate for their status and telemetry — the same `req_status`/`req_telemetry` calls the phone app and `meshcore-cli` make, through the companion Meshpoint already holds. MeshCore nodes advertise identity only (unlike Meshtastic's broadcast telemetry), so this fills the otherwise-empty repeater stats: battery, uptime, airtime, packet counters (flood/direct), noise floor, and LPP sensors (temp/humidity/pressure). Results show on a new **Repeaters** page (Radio sidebar group, visible only when polling is configured): per repeater, a **Health** card (battery, uptime, airtime, packet counters, noise floor, SNR, errors), a **Sensors** card listing every LPP reading it reports (all channels — temperature/humidity/pressure/altitude/solar), a **History** card (min/avg/max over stored telemetry), and a full-width **Trends** chart plotting the voltage, temperature, humidity and pressure history (rendered by the shared node-drawer chart, legend-toggleable; humidity and pressure were added to that chart so both the repeater trends and the Meshtastic node drawer gain them when a node reports them). The charted fields flow into the telemetry table for the node drawer and CSV export. Active two-way RF, so it's off by default, polls sequentially on a slow cadence, and only targets repeaters you have the login password for; passwords live in `local.yaml` and are never exposed by the API. A poll that fails transiently (the companion's command channel momentarily busy) is retried up to 3 times with a short gap before giving up. Zero-value sensor channels (an unused solar input at 0 V / 0 A / 0 W) are hidden from the Sensors card, and the cards read as a stats page — a full-width Health / Sensors / History row with a channel/value count chip and a Trends chart below. New `src/hardware/`-style `src/transmit/repeater_poller.py` + `GET /api/meshcore/repeaters`.
- **Repeater neighbours list is now clickable.** The Health card's "Neighbours" count opens a modal listing each reported neighbour (pubkey prefix, freshest-first by time since last heard, SNR), with names resolved against the known node roster where a pubkey prefix already matches — one batch query across all repeaters, not per-neighbour. A neighbour with a resolved name opens the standard node drawer on click.
- **Repeater card drops the redundant "polled Xm ago" footer.** The Repeaters summary bar already has an "Oldest poll" stat; the identical per-card line was pure duplication once there's only one repeater configured.
- **Live repeater neighbour polls now feed the roster, and each repeater card shows its own "Farthest neighbour."** Previously this data only ever reached `repeater_status.json` (map/neighbour-modal display), never the database — `scripts/import_contacts.py`'s manual `neighbours.json` import was the only way any of this ever reached the `nodes`/`packets` tables. Each poll now also creates/refreshes a placeholder node per reported neighbour (name unknown pubkeys, never overwrites an already-known name/role/position) and a synthetic `nb:<repeater_key>:<node_id>:<timestamp>` packet — same tagging convention as the manual import, just live and attributed to the specific repeater that reported it. `last_heard` only ever moves forward relative to a genuinely-captured packet (a stale secondhand report can never un-freshen a node heard more recently some other way), and nothing here inflates `packet_count` — it's not a packet Meshpoint's own radio received. The Repeaters page's Health card gains a **Farthest neighbour** row (distance, SNR, name) per repeater — measured from *that repeater's own position*, not Meshpoint's, since a polled repeater can be a remote site with a completely different local RF picture. These `nb:`-tagged rows are excluded from the Stats page's own "Farthest Direct Signal" (see the fix above), keeping the two concepts cleanly separated: what Meshpoint itself heard vs. what a repeater it operates heard.
- **Repeater polling gets a dashboard config page.** `repeater_poll:` was the last config section with actual secrets (repeater login passwords) that could only be set by hand-editing `local.yaml`. New **Configuration → Repeater Poll** page: enable/disable, poll interval, and add/rename/remove repeaters (up to 8) from the dashboard. Passwords follow the same convention as the MQTT broker password field — the API only ever reports whether one is *set*, never the value; leave the field blank when editing a repeater to keep its current password. New `PUT /api/config/repeater-poll`, and `GET /api/config` now includes a redacted `repeater_poll` summary. Changes require a restart to take effect, same as the other peripheral/capture-source config pages.
- **MeshCore packet feed no longer buried under bulk-imported telemetry history.** `scripts/import_meshcore_db.py`'s historical telemetry/status import intentionally writes one packet row per sensor channel (not a bug — a repeater's real LPP telemetry snapshot legitimately spans 10-20+ distinct channel/status fields), which meant every historical poll timestamp exploded into a wall of near-identical-looking `telemetry` rows in `GET /api/meshcore/packets`, drowning out genuinely recent mesh activity. That endpoint now excludes `capture_source='meshcore_db_import'` rows by default (`?include_imported=true` still exposes them if ever needed) — the underlying data isn't lost, it's just not the right thing to show in a "what's happening right now" feed; the repeater's own History/Sensors/Trends cards (powered by the separate `telemetry` table) are unaffected either way. Also added two maintenance scripts: `scripts/dedupe_meshcoredb_packets.py` (removes exact `packet_id` duplicates, dry-run by default) and `scripts/purge_meshcoredb_packets.py` (removes the telemetry/status clutter rows entirely, deliberately leaving `meshcoredb:neighbour:%` rows alone since the Topology page's graph query actively depends on them for historical neighbour-star edges).
- **Repeater login no longer gets permanently stuck on a stale route.** User report: a repeater that had been polling fine for a long time started failing every login attempt ("login failed or timed out") continuously — password confirmed correct (a separate BLE-connected tool logged into the same repeater successfully throughout), reception of everything else over that same USB companion kept working normally, and a full Pi reboot didn't help either, ruling out a wrong password or a wedged local session. A directed login is routed over the contact's cached `out_path` in the underlying `meshcore` library; if the mesh topology shifts after that path was learned (a relay it went through reboots, moves, or drops out), every future login keeps retrying the same dead route and times out identically forever — invisible to passive/flood-routed traffic (adverts, messages) since those don't depend on a stored path at all, and untouched by a Meshpoint-side reboot since the stale path is cached state, not a live connection. `poll_repeater()` (`src/transmit/meshcore_tx_client.py`) now calls the library's own `reset_path()` whenever a login fails, clearing the cached path so the very next retry (or next poll round) falls back to flood routing and can learn a fresh one; best-effort and silent on its own failure so it never masks the real login error being reported. **Follow-up fix, same day**: the first version of this fix shipped a bug that made it a no-op — `reset_path()` was called with the short 6-byte contact-lookup prefix (`key`), but the library's destination validation requires a full 32-byte public key (or a contact object carrying one) and raises `ValueError` on anything shorter; the broad `except Exception` around the reset call silently swallowed that, so the path was never actually reset and the live incident this was built for kept failing identically after deploying the "fix." Corrected to pass the already-resolved `contact` object (which carries the full `public_key`) instead of the bare prefix string. **Second follow-up fix, same day**: the corrected reset still wasn't covering every failure shape — it only ran inside the explicit `login is None` branch, but a `wait_for()` timing out anywhere in the login/status/telemetry/neighbours sequence (including login's own wait) raises `asyncio.TimeoutError` past that check entirely, landing in the function's outer exception handlers (`out["error"] = "timed out"`) where no reset ever ran. Moved the reset into the function's `finally` block, gated on any non-empty `out["error"]`, so it now fires uniformly regardless of which specific step or timeout produced the failure. **Diagnostic follow-up, same day**: live logs confirmed the reset now runs successfully on every failed attempt (no exception) but the underlying repeater still times out on every retry regardless — meaning a stale cached path alone doesn't fully explain this specific incident, or the reset isn't landing as expected on the companion device. The reset call's own result wasn't being inspected, only whether it raised, so there was no visibility into what the companion actually replied. Now logs the device's own response event (`OK`/`ERROR`/etc, same pattern already used for `send_advert`) alongside the existing "reset cached path" line, giving a concrete next signal to look at.

- **The service user now owns the whole install tree.** The fan-control fix chowned only the top of `/opt/meshpoint` to the service user, which flipped the sudo auto-detection above into running plain git against the still-root-owned `.git` — Check for updates failed with "cannot open '.git/FETCH_HEAD': Permission denied". Fixed with permissions, recursively: `install.sh`, the systemd unit (every start), and `post_update.sh` (every dashboard apply) all `chown -R` the entire tree to `meshpoint`, so the root-owned files that the apply chain's `sudo git fetch/reset` leaves behind are handed back before the next update check. The sudo auto-detection also probes `.git` (the directory fetch actually writes to) instead of the repo top, so it picks the right mode in mixed-ownership states.
- **Self-update now points at whatever fork you actually cloned, not a hardcoded maintainer's repo.** The version-check and raw-file URLs were hardcoded to `javastraat/meshpoint`; a user running their own fork had no way to redirect them short of editing source. New `src/remote/repo_source.py` reads the `origin` remote straight out of `.git/config` (no `git` subprocess) and derives `owner/repo` from it — a fork's self-update targets itself automatically, with zero configuration. Falls back to the upstream `KMX415/meshpoint` project (not any particular fork) when origin can't be parsed (missing remote, non-GitHub host, unreadable config). The Updates page's "Upstream" card now shows and links to the actual resolved repo instead of the generic `origin/<branch>` label. New `GET /api/update/branches` (admin-only, GitHub API, 5-minute cache) lists the resolved repo's real branches, so the "Custom branch" field now offers them as autocomplete suggestions instead of requiring the exact name typed blind.

### v0.7.3.1 (May 13, 2026)

Hotfix on top of v0.7.3 the same day. Reported by Willard on Discord ~3h after release: dashboard stuck on "Reconnecting..." with no data after upgrading. Two compounding bugs in the new auth path; a stale browser tab against an auth-required server is enough to trigger both.

- **WS auth close frame now actually reaches the browser.** `src/api/server.py` was calling `await websocket.close(code=4401)` *before* `await websocket.accept()`, which causes Starlette to fail the WebSocket handshake with HTTP 403 instead of completing the handshake and sending a close frame. Browsers translate that to close code `1006` (Abnormal Closure) on the JS side, and our `frontend/js/websocket_client.js` only special-cases `4401` for the redirect-to-/login path -- so unauthenticated WS connections fell through to the generic reconnect loop and stuck forever. Fix: `accept()` first, then `close(code=WS_AUTH_CLOSE_CODE)`. Validated end-to-end on .141 (cookie cleared, dashboard refreshed → bounces to /login as intended).
- **Dashboard root (`/`) now redirects unauthenticated requests to `/login`.** The `StaticFiles(directory=..., html=True)` mount on `/` was serving `index.html` to everyone with no auth check, so a stale browser tab could load the new SPA JS and immediately fight the now-auth-required `/ws`. New explicit `@app.get("/")` route registered ahead of the static mount: 302s unauthenticated requests to `/login` (or `/setup` if no admin password is set yet), serves `index.html` for valid sessions. Static asset paths (`/css`, `/js`, `/assets`, etc.) still fall through to the existing mount.
- **Client defense in depth.** `frontend/js/websocket_client.js` now tracks whether the socket reached the `open` state. If `onclose` fires *without* a prior `onopen`, the handshake failed before the close frame could be delivered. The client probes an auth-gated endpoint (`/api/device/status`); the global 401 interceptor in `app.js` then handles the redirect if it's an auth-shaped failure, while real network blips fall through to the existing reconnect schedule. Belt-and-suspenders coverage so a future server-side regression in the close-code path can't strand users again.
- **Internal:** new `tests/test_dashboard_root_route.py` (unauthenticated GET `/` → 302 to `/login`; GET `/` with no admin password yet → 302 to `/setup`; GET `/` with valid cookie → 200 + index.html). New `tests/test_websocket_auth_close_code.py` asserts the WS handshake completes and the close frame's code is exactly `4401` for both no-cookie and bad-cookie cases (regression for the `accept()`-before-`close()` requirement). 403 tests passing, ruff clean.

- **Drop `STAT_CRC_BAD` packets at the HAL boundary instead of forwarding them to the decoder.** `SX1302Wrapper.receive()` was logging the diagnostic WARNING for CRC-failed packets but still appending them to the returned packet list, with `crc_ok=False` set on the `ConcentratorPacket`. Hardware-validated on RAK V2: 14 historical phantoms in the local DB matched the bit-flip fingerprint of legitimate neighbors (`7d8b98a9`, `a0dd8936`, etc.), and zero new phantoms have entered the database since the fix. ### v0.6.6 (April 25, 2026)

MeshCore reliability patch. Small follow-up to v0.6.5 cleaning up rough edges around the MeshCore USB companion. No edge concentrator changes, no cloud changes.

- **Companion connects cleanly on `systemctl restart meshpoint`.** ESP32-S3 boards (Heltec V3/V4 etc.) need 6-10 seconds to be USB-ready after a reboot, but the underlying meshcore library was giving up after 5. Whether this was happening on other Meshpoints in production is unknown; it was never surfaced as a user-visible symptom.
