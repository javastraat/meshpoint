# v0.7.7: Mesh broadcast cadence UI (position + telemetry)

**Status:** Implemented on `feat/v0.7.7-mesh-broadcast-cadence`. **GitHub:** [#92](https://github.com/KMX415/meshpoint/issues/92). **Branch:** `feat/v0.7.7-mesh-broadcast-cadence` from `feat/v0.7.7`. **Vehicle:** edge-only patch on the v0.7.7 sprint (pure Python, no concentrator recompile, no cloud).

**Trigger:** Discord report from Chiumanfu (2026-07): NodeInfo set to 24h but `POSITION_APP` every 15 min and `TELEMETRY_APP` every 30 min still visible on witness nodes. Root cause is three independent broadcast loops, only NodeInfo has dashboard controls today.

---

## Problem

| Broadcaster | Yaml key | Default | Dashboard today |
|-------------|----------|---------|-----------------|
| NodeInfo | `transmit.nodeinfo.interval_minutes` | 180 min | Configuration → Radio (chips + hot-reload) |
| Position | `transmit.position.interval_minutes` | **15 min** | **None** (yaml only) |
| Telemetry | `transmit.telemetry.interval_minutes` | **30 min** | **None** (yaml only) |

Configuration → GPS exposes mesh position **source** (registered pin vs live GPS, privacy) but not **cadence**. Users reasonably assume the NodeInfo interval governs all periodic mesh TX.

---

## Goals

1. Dashboard controls for position and telemetry broadcast intervals (0 = off, 5–1440 min).
2. Hot-reload on save (match NodeInfo UX: no restart for interval changes).
3. Live status on Configuration → Radio: next due / last sent / running (parity with NodeInfo countdown card).
4. Docs + CHANGELOG so support can point users at the right cards.

**Non-goals (this sprint):**

- `Send Now` for position/telemetry (stretch only; NodeInfo already has it).
- Changing defaults (15 / 30 min stay; users opt into slower cadence).
- MQTT or upstream cadence (separate surfaces).
- SX1261 MeshCore RX work (`docs/plans/v0.7.7-sx1261-meshcore-rx.md` stays independent).

## Architecture

### Backend: shared interval hot-reload

`NodeInfoBroadcaster` already implements the canonical loop (`asyncio.Event` wake, `set_interval()`, `last_sent_at` / `next_due_at`). `PositionBroadcaster` and `TelemetryBroadcaster` still use a naive `sleep(interval)` loop with no hot-reload.

**Recommended approach:** extract a small reusable helper (single file, one responsibility) rather than subclassing three broadcasters:

```
src/transmit/broadcast_interval.py   # NEW (~80 lines)
  - clamp_interval_minutes()          # move from nodeinfo_broadcaster (re-export for compat)
  - BroadcastIntervalState            # holds interval seconds, Event, last_sent_at, startup anchor
  - sleep_until_due(state)            # async helper used by all three loops
  - apply_interval_change(state, minutes) -> clamped
```

Then refactor:

- `nodeinfo_broadcaster.py` — adopt helper (behavior unchanged; existing tests must stay green).
- `position_broadcaster.py` — event-based loop + `set_interval()` + telemetry properties.
- `telemetry_broadcaster.py` — same.

Properties each broadcaster exposes for API status blocks:

- `is_running`, `last_sent_at`, `next_due_at`, `interval_seconds`

**Enable/disable at runtime:**

| Change | Behavior |
|--------|----------|
| `interval_minutes: 0` while running | Pause loop (same as NodeInfo) |
| `0 → N` while running | Resume; next TX at `last_sent_at + N` or immediately if overdue |
| `N → 0` at boot | Broadcaster never starts (existing `_build_*` guard) |
| `0 → N` at boot when TX disabled | `restart_required: true` in API response (mirror nodeinfo) |

`startup_delay_seconds` changes still require restart (document in UI hint).

### API routes

Mirror `src/api/routes/nodeinfo_routes.py`:

| Endpoint | Purpose |
|----------|---------|
| `PUT /api/config/position` | `interval_minutes`, optional `startup_delay_seconds` |
| `PUT /api/config/telemetry` | same |
| `GET /api/config` | add `position` and `telemetry` top-level blocks with live status |

New files (keep under 500 lines each):

- `src/api/routes/position_broadcast_routes.py`
- `src/api/routes/telemetry_broadcast_routes.py`

Wire in `server.py`:

- `position_broadcast_routes.init_routes(config, position_broadcaster)`
- `telemetry_broadcast_routes.init_routes(config, telemetry_broadcaster)`
- Register routers on FastAPI app (same pattern as nodeinfo).

`config_enrichment.py`: add `transmit.telemetry` to GET payload (position interval already partially exposed; ensure symmetry).

Persist via `save_section_to_yaml("transmit", {"position": {...}})` / `{"telemetry": {...}}` — merge-friendly partial writes like nodeinfo.

### Frontend

**Shared component** (avoid triplicating chip UI):

```
frontend/js/configuration/broadcast_interval_card.js   # NEW
  - class BroadcastIntervalCard { presets, mount, render, save }
  - constructor(api, { title, hint, saveLabel, putUrl, configKey })
```

**Wrappers** (thin, mount-only):

- `position_broadcast_card.js` — GPS section, `putUrl: '/api/config/position'`, `configKey: 'position'`
- `telemetry_broadcast_card.js` — Radio section, `putUrl: '/api/config/telemetry'`, `configKey: 'telemetry'`

**Orchestrator edits:**

- `configuration_panel.js` — mount telemetry card under Radio; mount position card inside `GpsConfigCard` mount or as sibling in GPS panel.
- `index.html` — script tags for new JS files.

**Status strips (optional P1, can ship in same PR):**

- Clone `RadioNodeInfoCard` pattern → `RadioTelemetryStatusCard` reading `config.telemetry.running / next_due_at`.
- Position status can be a single line under the interval fieldset (avoid a third full countdown card unless time allows).

CSS: reuse existing `cfg-chip`, `cfg-chip--off`, `cfg-btn--has-pending` from NodeInfo card. No new tokens.

## Hardware validation (.141)

| Step | Expected |
|------|----------|
| Default boot | journal: position 180s startup / 900s interval; telemetry 120s / 1800s |

| NodeInfo still **24h** | Unchanged; proves independence |
