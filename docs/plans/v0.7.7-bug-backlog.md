## Private channel messages show under LongFast (dashboard save path)

**GitHub:** [#89](https://github.com/KMX415/meshpoint/issues/89)

**Symptom (v0.7.6):** User adds a private Meshtastic channel via Configuration → Channels. Outbound from Meshpoint decrypts and displays on peers correctly. Inbound private-channel broadcasts on the same Meshpoint appear in the **LongFast** (`broadcast:meshtastic:0`) conversation even though decryption succeeds (custom 256-bit key).

**Root cause:** `channel_hash_map` in `src/api/server.py` `_setup_message_interception` is built **once at startup**. `PUT /api/config/channels` updates `CryptoService` keys live (`config_routes.update_channels`) but does **not** rebuild the map. Inbound routing uses `channel_hash_map.get(packet.channel_hash, 0)`, so unknown hashes fall through to channel index `0` (LongFast). TX path is unaffected (`TxService._resolve_channel` reads live crypto state).

**Workaround:** `sudo systemctl restart meshpoint` after saving channels on the dashboard.

**Validation (2026-06-11):** Gus confirmed restart fixed inbound routing on both Meshpoints (private channel messages no longer land in LongFast). Matches the stale `channel_hash_map` hypothesis.

**Fix shape (not started):**

- Extract `ChannelHashResolver` (rebuild + lookup) from the message-interception closure.
- Call `rebuild()` from `update_channels` after crypto refresh.
- Optional: WARN log once per unmapped `channel_hash`.
- Tests: rebuild after `PUT /channels`, inbound routes to correct `broadcast:meshtastic:N`.

**Files:** `src/api/server.py`, `src/api/routes/config_routes.py`, new resolver module, tests.

**Edge-only.** No cloud impact.
